## このページの役割

このページは、`Web技術` と `クラウドコンピューティング` を整理する知識ノードです。`HTTP/HTTPS` から始まる Web の基礎、`API` と `Web サービス` の設計パターン、そして `クラウドサービスモデル` と `デプロイメント形態` を一本の流れで押さえます。

## 学習のポイント

- まず `HTTP/HTTPS` と `cookie / セッション管理` で Web 通信の基本を理解する
- 次に `API` の設計パターン（`REST` vs `SOAP` vs `GraphQL`）と `Web サービス` を分ける
- そのうえで `クラウドの 3 層モデル`（`SaaS / PaaS / IaaS`）と `デプロイメント形態` を整理する
- 最後に `仮想化`、`コンテナ`、`サーバレス` などの実装技術をスタック図で押さえる

## 試験で何が問われるか

- `HTTP/HTTPS` と `cookie / セッション` の役割を説明できるか
- `REST` の特徴（ステートレス、リソース指向）を言えるか
- `SaaS / PaaS / IaaS` を、提供範囲と管理責任で区別できるか
- `仮想マシン` と `コンテナ` の違いを実装と効率の観点で説明できるか
- `オンプレミス` と `クラウド` の使い分けを、コストと柔軟性で読めるか
- `パブリック / プライベート / ハイブリッド` クラウドの使い分けを言えるか

## 定義と仕組み

### HTTP と HTTPS

`HTTP` は Web ブラウザとサーバ間でテキストやファイルをやり取りするプロトコルです。`HTTPS` は、その通信を TLS/SSL 暗号化で保護します。試験では `ステートレス`（各リクエストが独立）という特性が重要です。

### Cookie とセッション管理

`HTTP` はステートレスなため、ユーザーをどう識別し続けるかが課題です。`Cookie` はクライアント側に小さなデータを保存し、毎回のリクエストで自動送信する仕組みです。`セッション` はサーバ側でユーザーの状態を保持し、Session ID を Cookie で受け渡す仕組みです。試験では、認証後の状態保持がどう実装されるかを理解することが大切です。

### Ajax と WebSocket

`Ajax` は、ページ全体をリロードせず、JavaScript で部分的にサーバと通信する技術です。`WebSocket` は、HTTP をアップグレードして双方向通信を実現するプロトコルです。リアルタイム性が必要なチャットやライブ更新では WebSocket、ページ部分更新では Ajax という使い分けが基本です。

### PWA（Progressive Web App）

`PWA` は、ブラウザ上で App 同様の体験（オフライン動作、プッシュ通知、ホーム画面インストール）を提供する Web アプリケーションです。Service Worker というブラウザバックグラウンドプロセスがキャッシング戦略を管理します。試験では、`Web アプリの進化形` として位置付けます。

### REST と Web API

`REST`（Representational State Transfer）は、Web 上のリソース（URI）に対して、HTTP メソッド（GET, POST, PUT, DELETE）で操作する設計原則です。`ステートレス`、`リソース指向`、`キャッシュ可能` が特徴です。試験では、RPC 的な設計との違いを理解することが重要です。

### SOAP と Web Service

`SOAP` は、XML ベースの複雑な交換プロトコルです。`WSDL`（Web Services Description Language）で仕様を定義し、`UDDI` で検索可能にするエンタープライズ向けの仕組みです。REST が単純な CRUD 中心なのに対し、SOAP はトランザクション性が求められるシステム間連携に使われます。

### GraphQL

`GraphQL` は、クライアント側で必要なデータ構造を指定して取得する言語です。REST のような複数エンドポイント呼び出しや、不要なデータ取得の無駄を削減できます。試験では、REST との設計思想の違いを認識することが大切です。

### JSON と XML

`JSON`（JavaScript Object Notation）は、キーと値のペアをネストできる軽量な形式です。`XML` は、タグ構造で階層的なデータを表現できる拡張可能形式です。API 設計では、JSON がシンプルさで、XML が複雑なスキーマ定義で使い分けられます。

### マイクロサービス vs モノリス

`モノリス` アーキテクチャは、全機能が 1 つの大きな成果物にまとまった設計です。`マイクロサービス` は、ビジネス機能ごとに小さなサービスに分割し、API で疎結合する設計です。スケーリング性、デプロイ独立性、障害隔離の観点でマイクロサービスが優位ですが、運用複雑性が増します。

## クラウドコンピューティングの概要

クラウドコンピューティングは、インターネット経由で計算リソース（サーバ、ストレージ、ネットワーク、ソフトウェア）をオンデマンド提供する仕組みです。試験では、`サービスモデル` の 3 層と `デプロイメント形態` の違いを中心に理解します。

### SaaS / PaaS / IaaS の 3 層モデル

| サービス | 意味 | 提供範囲 | ユーザー管理範囲 | 典型例 |
| --- | --- | --- | --- | --- |
| `SaaS` | Software as a Service | アプリケーション | なし（すべてクラウド企業） | Google Workspace, Salesforce, Microsoft 365 |
| `PaaS` | Platform as a Service | ミドルウェア + 開発環境 | アプリケーション | Heroku, Google App Engine, Azure App Service |
| `IaaS` | Infrastructure as a Service | 計算リソース（CPU、メモリ、ストレージ） | OS、ミドルウェア、アプリケーション | AWS EC2, Azure VM, Google Compute Engine |

各層は、ユーザーが自分で管理する部分が異なります。SaaS はブラウザで使うだけ、PaaS はコードを書いてアップロード、IaaS はマシンを借りて自分でセットアップします。試験では、`自分で何をやるのか` という観点で区別することが大切です。

### FaaS（Function as a Service）

`FaaS` は、関数単位で実行できるサービスです（AWS Lambda など）。`サーバレス` と呼ばれることもあります。実行時間とメモリ使用量だけを課金する従量制が特徴です。スケーリングも自動です。

### デプロイメントモデル

| 形態 | 特徴 | 管理 | セキュリティ | コスト | 用途 |
| --- | --- | --- | --- | --- | --- |
| `パブリック` | 複数の顧客で共有 | クラウド企業 | 標準対応 | 安い | Web サービス、開発環境 |
| `プライベート` | 1 企業専有 | 企業またはベンダー | カスタマイズ可能 | 高い | 機密データ、レガシーシステム |
| `ハイブリッド` | オンプレミス + クラウド | 混合 | 混合 | 中程度 | 段階的移行、ピーク対応 |
| `コミュニティ` | 特定業界など共通団体 | 共通 | 業界標準 | 中程度 | 業界特有の規制対応 |

ハイブリッドクラウドは、既存オンプレミス資産と新規クラウドサービスを組み合わせる実務的な選択です。マルチクラウドは、複数のクラウド企業（AWS、Azure、Google Cloud など）を組み合わせる戦略です。

### クラウド関連技術スタック

クラウドの効率性を支える実装技術を層で理解します。

#### 仮想化

`仮想化` は、物理的な 1 台のサーバ上に、複数の独立した仮想マシンを走らせる技術です。CPU、メモリ、ディスクを論理的に分割し、各 VM に割り当てます。ハイパーバイザーが、物理リソースと VM 間のマッピングを管理します。

#### 仮想マシン vs コンテナ

| 項目 | 仮想マシン | コンテナ |
| --- | --- | --- |
| 単位 | 完全な OS を含む | アプリケーション + 依存ライブラリ |
| 起動時間 | 数十秒～分 | 数秒以下 |
| リソース効率 | 1 VM あたり GB 単位のメモリ | 1 コンテナあたり MB 単位 |
| 隔離レベル | ハード（異なる OS も可） | ソフト（同じ OS カーネル共有） |
| 移植性 | ハイパーバイザー依存 | Docker イメージなら環境非依存 |
| ディスク容量 | 数 GB～ | 数百 MB～ |
| 実装例 | VMware vSphere, Hyper-V | Docker, Kubernetes |

仮想マシンは、企業サーバ統合やマルチテナント環境で使われます。コンテナは、マイクロサービスやクラウドネイティブアプリケーションで使われます。試験では、`何を分離するか`、`どれだけリソースを消費するか` の観点で見分けることが重要です。

#### コンテナオーケストレーション

`Docker` はコンテナ化の実装技術です。`Kubernetes`（K8s）は、複数のコンテナを複数のマシンで実行・管理するオーケストレーションプラットフォームです。スケーリング、ヘルスチェック、ローリングアップデートなどを自動化します。

#### CDN（Content Delivery Network）

`CDN` は、地理的に分散したサーバで静的コンテンツ（画像、動画、JS、CSS）をキャッシュ配信する仕組みです。ユーザーに近いサーバから配信するため、遅延が減り、オリジンサーバの負荷も軽減されます。

#### エッジコンピューティング

`エッジコンピューティング` は、クラウド中央ではなく、ネットワークの末端（エッジ）でデータ処理を行う考え方です。IoT センサからのデータをエッジで集約・フィルタリングして、オリジンへ送る仕組みです。遅延削減とプライバシー保護が利点です。

#### サーバレスコンピューティング

`サーバレス` は、開発者がサーバ管理を気にせず、関数やイベントハンドラを書くだけで動く環境です。AWS Lambda、Google Cloud Functions、Azure Functions が該当します。スケーリングと課金が自動で、小粒度で独立した処理に向いています。

#### VDI（Virtual Desktop Infrastructure）

`VDI` は、ユーザーが使うデスクトップ環境そのものをサーバ上で仮想化し、ネットワーク経由でシンクライアントへ配信する仕組みです。テレワークや BYOD 環境でセキュリティを保ちながら柔軟に働ける利点があります。

### オンプレミス vs クラウド

| 観点 | オンプレミス | クラウド |
| --- | --- | --- |
| 初期投資 | 大（ハードウェア購入） | 小（月額課金） |
| 運用コスト | 高（人員、電力、冷却） | 中程度 |
| カスタマイズ性 | 高い | サービス依存 |
| セキュリティ管理 | 自社責任 | 共有責任 |
| スケーラビリティ | 制限（物理的上限） | 無制限（ほぼ） |
| 初期構築期間 | 数ヶ月～ | 数日～ |
| 法規制対応 | 自社で確認・実装 | サービス提供企業が対応 |

オンプレミスは、機密性が高い情報や、規制が厳しい業界（金融、医療）で適切です。クラウドは、成長性の高いスタートアップや、ピーク時の負荷対応が必要な企業に向いています。ハイブリッドは、両者の長所を活かす実務的な選択です。

## 典型例

| 論点 | 典型的な見分け方 |
| --- | --- |
| `HTTP` と `HTTPS` | 通信の暗号化が必要か |
| `Cookie` と `セッション` | クライアント側に持つか、サーバ側に持つか |
| `Ajax` と `WebSocket` | ページ部分更新か、リアルタイム双方向か |
| `REST` と `SOAP` | シンプルな CRUD か、複雑なトランザクションか |
| `REST` と `GraphQL` | 固定エンドポイントか、クライアント指定データか |
| `SaaS` と `PaaS` | アプリそのものか、アプリを作る環境か |
| `PaaS` と `IaaS` | 開発環境か、インフラそのものか |
| `仮想マシン` と `コンテナ` | OS を含めるか、アプリだけか |
| `オンプレミス` と `クラウド` | 完全自社管理か、従量制か |
| `パブリック` と `プライベート` | 複数企業共有か、専有か |

## 典型的なつまずき

- `HTTP` と `HTTPS` の違いを、ただ `セキュア / 非セキュア` と覚えてしまう
- `Cookie` と `セッション` を同じものだと思い込む
- `REST` を単なる `API 設計` だと捉え、`ステートレス` の特性を見落とす
- `SaaS / PaaS / IaaS` を語感だけで覚え、`ユーザーが管理する部分` を理解しない
- `仮想マシン` と `コンテナ` を、どちらも `複数プロセスを走らせるもの` と同一視する
- `クラウド` を `常に安い` と思い込み、オンプレミスとのトータルコストを検討しない
- `マイクロサービス` を便利な魔法だと思い、複雑な運用コストを無視する
- `サーバレス` を `サーバがない` と言葉どおり解釈し、スケーリング上限を見落とす

## 問題を解くときの観点

- 問われているのは `Web プロトコル`、`API 設計`、`クラウドモデル`、`実装技術` のどれか
- `責任の線引き` が問題の重要なポイント（SaaS/PaaS/IaaS の使い分けここ）
- セキュリティやコンプライアンスが条件なら、パブリック / プライベート / ハイブリッドの選択に影響するか
- `スケーラビリティ` や `可用性` が求められるなら、オンプレミスとクラウドのどちらが適切か
- `初期投資` と `運用コスト` のトレードオフを意識しているか
- 技術選択が、顧客体験やビジネスモデルにどう影響するか

## 確認問題

<Callout title="確認問題 1：SaaS / PaaS / IaaS の判定">
  以下の 3 つのサービスを SaaS / PaaS / IaaS に分類し、それぞれについてユーザーが管理する範囲を説明してください。

  1. Google Workspace（メール、ドキュメント作成など）
  2. Heroku（Web アプリケーションのデプロイ基盤）
  3. AWS EC2（仮想マシンレンタルサービス）

  **想定される答え**
  - Google Workspace：SaaS（ユーザー管理なし、すべてクラウド企業）
  - Heroku：PaaS（アプリケーションコード、ユーザーが管理）
  - AWS EC2：IaaS（OS 以上、ユーザーが管理）
</Callout>

<Callout title="確認問題 2：REST の特徴">
  REST API の設計原則について、以下の 3 つの特性を説明してください。また、SOAP と比較して REST が選ばれる理由を述べてください。

  1. ステートレス
  2. リソース指向
  3. HTTP メソッド（GET, POST, PUT, DELETE）の使い分け

  **想定される答え**
  - ステートレス：各リクエストが独立し、サーバが状態を保持しない（スケーリング性が高い）
  - リソース指向：URI でリソースを表現し、その上で CRUD 操作を行う
  - HTTP メソッド：GET は取得、POST は作成、PUT は更新、DELETE は削除に対応
  - SOAP との比較：SOAP は複雑なトランザクション向け、REST は軽量な Web API 向けで、シンプルさとスケーラビリティで REST が優位
</Callout>

<Callout title="確認問題 3：仮想マシン vs コンテナ">
  仮想マシンとコンテナの実装上の違いを、以下の観点から説明してください。

  1. 含まれるもの（何を仮想化するか）
  2. 起動時間とリソース効率
  3. それぞれの用途

  また、マイクロサービスアーキテクチャでコンテナが選ばれる理由を述べてください。

  **想定される答え**
  - 仮想マシン：OS を含めて仮想化し、完全に独立した環境を提供（起動時間：数十秒～分、リソース：GB 単位）
  - コンテナ：アプリケーションと依存ライブラリだけをパッケージ化（起動時間：数秒以下、リソース：MB 単位）
  - 用途：VM はサーバ統合、マルチテナント；コンテナはマイクロサービス、クラウドネイティブ
  - マイクロサービスでコンテナが選ばれる理由：軽量、起動が速く、スケーリング性が高く、デプロイ頻度が上げやすい
</Callout>
