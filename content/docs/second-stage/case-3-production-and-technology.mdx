## この事例の位置づけ

事例Ⅲは、`改善っぽい言葉` を並べる試験ではなく、`どの工程で / なぜ詰まり / どう直し / QCD の何が良くなるか` を助言する試験です。現場で回る運用改善に落とし込めるかが中心になります。

<Callout title="この事例の位置づけ">
  現場の問題を工程に沿って特定し、`どの改善策で`、`QCD のどこが`、`どう良くなるか` を具体的に書くことが重要です。抽象論より、現場で実際に回る運用改善へ落とし込めるかが問われます。
</Callout>

> [!TIP]
> 初めて読むならこの順
>
> まず [事例Ⅲの設問型](/second-stage/case-3-question-patterns) で `どの工程のどの問題か` を切ります。そのあとに `計画と統制 → 動線とレイアウト → 品質と標準化 → 作業改善 → 日程とボトルネック → 設備と外注` の順で 1 次知識を補い、最後に [事例Ⅲの答案骨子](/second-stage/case-3-answer-frameworks) へ戻って `問題 → 原因 → 改善策 → QCD効果` の短文へ落とすと、抽象語だけの答案になりにくくなります。

## 学習のポイント

- 現状の問題点を工程に沿って整理し、改善策を QCD に結び付ける
- `問題点` と `課題` を分け、現状分析と将来助言を混同しない
- 標準化、段取り改善、レイアウト、情報共有、外注管理を具体策として使えるようにする
- 抽象的な提案ではなく、現場で回る運用レベルまで落とす

## 頻出の設問パターン

- 現状の生産管理上の問題点を整理させる問題
- 生産計画、生産統制、進捗管理、納期管理を問う問題
- 品質向上、段取り短縮、在庫削減、標準化を問う問題
- レイアウト、設備、情報共有、IT 活用を問う問題
- 新規受注や新市場対応に向けた生産体制整備を問う問題

## 問題を解くときの観点

<Steps>
  <Step>
    ### 詰まる工程を先に特定する

    `どの工程で問題が起きているか` を切って、全体論だけの答案を防ぎます。
  </Step>
  <Step>
    ### 原因を 4M で整理する

    `人、設備、方法、材料` のどこに原因があるかを見て、改善策の方向を決めます。
  </Step>
  <Step>
    ### 現場運用として書く

    標準化、段取り、進捗管理、見える化など、現場で回る改善策に落とし込みます。
  </Step>
  <Step>
    ### QCD への効果で締める

    品質向上、コスト低減、納期短縮のどこに効くかを最後に示して、助言を閉じます。
  </Step>
</Steps>

## よく使う助言の方向

<Cards>
  <Card title="標準化と教育">
    標準化、手順書整備、教育で作業ばらつきを減らし、品質を安定させます。
  </Card>
  <Card title="計画と進捗管理">
    生産計画と進捗管理を見直し、負荷平準化と納期管理を強めます。
  </Card>
  <Card title="段取りとレイアウト">
    段取り改善、レイアウト見直し、動線短縮でムダを減らします。
  </Card>
  <Card title="見える化と情報共有">
    情報共有と見える化を進め、現場判断と連携速度を上げます。
  </Card>
  <Card title="外注と工程間連携">
    外注先管理や工程間連携を整え、部分最適ではなく全体最適へつなげます。
  </Card>
</Cards>

## 典型的なつまずき

- `効率化する` のような抽象語だけで、何をどう変えるかが書けない
- 問題点と改善策が対応していない
- QCD のどこに効く施策なのかが見えない
- 設備導入や IT 導入を書くだけで、標準化や教育が抜ける
- 現場改善の事例なのに、経営理念の話へ逃げてしまう

## よくある失点パターン

| 失点パターン | 起こりやすい原因 | 解く前の確認 |
| --- | --- | --- |
| `問題` と `課題` を混同する | 設問要求の言葉を分けていない | 現状の悪い事象か、目標かを決める |
| テンプレだけで書く | 与件の工程や制約を拾えていない | 工程順に問題を 1 回並べ直す |
| 改善策が抽象的 | 4M で具体化していない | `人、設備、方法、材料` のどこを動かすか書く |
| 生産面だけで閉じる | 最終設問の戦略性を見ていない | 営業や市場対応まで求められていないか確認する |
