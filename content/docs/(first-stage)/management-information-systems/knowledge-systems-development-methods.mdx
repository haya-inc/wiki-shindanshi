## このページの役割

このページは、`システムをどう作るか` を整理する知識ノードです。`共通フレーム`、`要件定義`、`ウォーターフォール`、`V字モデル`、`アジャイル`、`DevOps` を、工程と変更への強さで比較します。

## 学習のポイント

- まず `共通フレーム` で工程の全体像をつかむ
- 次に `要件定義` と `設計` の違いを切る
- そのうえで `ウォーターフォール`、`V字モデル`、`アジャイル`、`DevOps` を比較する

## 試験で何が問われるか

- `共通フレーム` の工程を、企画、開発、運用の流れで説明できるか
- `要件定義` と `設計` の違いを言えるか
- `ウォーターフォール`、`V字モデル`、`アジャイル` を、変更への強さや進め方で比較できるか
- `スクラム`、`DevOps` を、反復開発と運用連携の文脈で読めるか

## 定義と仕組み

### 共通フレームと工程

`共通フレーム` は、情報システムの企画、要件定義、開発、運用を共通の言葉で整理する枠組みです。試験では、`どの工程の話か` を先に決めると、要件定義と設計、開発と運用を混ぜにくくなります。

### 要件定義と設計

`要件定義` は、`何を作るか` を決める工程です。`設計` は、`どう作るか` を決める工程です。利用者が求める機能や制約を固めるのが要件定義、処理方式や画面、データ構造へ落とすのが設計と整理します。

### ウォーターフォールと V字モデル

`ウォーターフォール` は、工程を順に流す進め方です。`V字モデル` は、その流れに `各工程に対応したテスト` を対応付けた見方です。したがって、V字モデルは `テストとの対応` を強調した整理だと読むと分かりやすくなります。

### アジャイル、スクラム、DevOps

`アジャイル` は、短い反復で価値を確認しながら進める考え方です。`スクラム` は、その代表的な実践枠組みです。`DevOps` は、開発と運用を分断せず、継続的に改善する考え方です。試験では、`変更に強い`、`短い反復`、`開発と運用の連携` が切り分けの軸です。

## 典型例

| 論点 | 典型的な見分け方 |
| --- | --- |
| `要件定義` と `設計` | 何を作るかの合意か、どう作るかの具体化か |
| `ウォーターフォール` と `アジャイル` | 工程を順に固めるか、反復しながら進めるか |
| `ウォーターフォール` と `V字モデル` | 単純な工程順か、工程とテストの対応付けか |
| `スクラム` と `DevOps` | 開発の進め方か、開発と運用の連携か |

## 典型的なつまずき

- `要件定義` と `設計` をどちらも `仕様を決める工程` として混ぜる
- `ウォーターフォール` と `V字モデル` を別物の開発方法だと思い込む
- `アジャイル`、`スクラム`、`DevOps` を全部同じ粒度の言葉として覚える
- 開発手法の比較を、`新しい / 古い` だけで見てしまう

## 問題を解くときの観点

- 問われているのは `工程の位置`、`進め方`、`テスト対応`、`連携の仕組み` のどれか
- 要求変更が多いのか、最初に固めやすいのか
- テストまで含めて整理しているか
- 開発だけの話か、運用まで含めた話か
