## このページの役割

このページは、`作ったシステムをどう安定運用し、どう評価するか` を整理する知識ノードです。`ITIL`、`SLA`、`システム監査`、`内部統制`、`信頼性指標` を、運用、評価、統制の流れで切り分けます。

## 学習のポイント

- まず `運用管理` と `評価・監査` を分ける
- 次に `ITIL` と `SLA` を、現場運用と約束事の違いで切る
- そのうえで `システム監査`、`内部統制`、`信頼性指標` を整理する

## 試験で何が問われるか

- `ITIL` が何を整理する枠組みかを説明できるか
- `SLA` を、利用者との合意水準として理解しているか
- `システム監査` と `内部統制` の違いを言えるか
- `可用性`、`MTBF`、`MTTR`、`RASIS` を、何を良くする指標かで読めるか

## 定義と仕組み

### 運用管理とサービスレベル

運用管理では、障害対応、変更管理、構成管理、サービスレベル管理のように、`止めない`、`品質を保つ` ための活動が中心になります。試験では、開発が終わったあとの `回す段階` の話だと切ることが重要です。

### ITIL と SLA

`ITIL` は、IT サービス運用を体系立てて整理する枠組みです。`SLA` は、提供側と利用側で `どの水準までサービスを保つか` を決める合意です。`ITIL` は運用管理の知識体系、`SLA` は合意された目標水準と読むと整理しやすくなります。

### システム監査と内部統制

`システム監査` は、情報システムが有効かつ安全に運用されているかを第三者的に点検する考え方です。`内部統制` は、業務や財務報告を適正に行うための統制です。前者は `点検`、後者は `日常の統制の仕組み` と分けます。

### 信頼性指標

`可用性` は使える状態をどれだけ保てるか、`MTBF` は故障と故障の間隔、`MTTR` は復旧までの時間を表します。`RASIS` は、信頼性や保守性のような評価軸をまとめて見る枠組みです。

## 典型例

| 論点 | 典型的な見分け方 |
| --- | --- |
| `ITIL` と `SLA` | 運用管理の知識体系か、合意したサービス水準か |
| `システム監査` と `内部統制` | 点検する立場か、日常的に統制する仕組みか |
| `MTBF` と `MTTR` | 壊れにくさか、直しやすさか |
| `可用性` と `信頼性` | 使える状態か、故障しにくさか |

## 典型的なつまずき

- `ITIL` をツール名や製品名だと思ってしまう
- `SLA` を運用管理全体と混同する
- `システム監査` と `内部統制` をどちらも `チェック` として同じに見てしまう
- `MTBF`、`MTTR`、`可用性` を全部 `止まりにくさ` として雑に覚える

## 問題を解くときの観点

- 問われているのは `運用`、`合意水準`、`点検`、`統制`、`評価指標` のどれか
- その指標は `壊れにくさ` と `復旧しやすさ` のどちらを見るか
- 利用者との約束の話か、内部で管理する話か
- 開発時の管理ではなく、運用段階の話か
