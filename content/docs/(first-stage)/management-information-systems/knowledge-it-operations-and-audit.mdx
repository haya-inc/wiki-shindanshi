## ITIL の基本構造

### ITIL とは何か

ITIL（IT Infrastructure Library）は、IT サービス運用を体系立てて整理した **ベストプラクティス集** です。ツールや製品ではなく、「運用をどう組織するか」という **知識体系** です。

**試験での頻出ポイント**: 「ITIL は枠組み、SLA は目標値」と切り分けることが重要。

### ITIL v3 の 5 つのライフサイクル

ITIL v3 では、IT サービスの全体を 5 つの段階に分けます。

| ライフサイクル | 主な目的 | 代表的プロセス |
| --- | --- | --- |
| **サービスストラテジ** | 事業目標に合わせた IT 戦略を立てる | サービスポートフォリオ管理、需要管理、財務管理 |
| **サービスデザイン** | サービスの構成と仕様を設計する | IT サービス継続性管理、キャパシティ管理、可用性管理 |
| **サービストランジション** | 開発したシステムを本番に移行する | 変更管理、リリース管理、構成管理 |
| **サービスオペレーション** | 日々のシステムを安定稼働させる | インシデント管理、問題管理、イベント管理 |
| **継続的サービス改善** | 運用の質を継続的に向上させる | サービスレベル管理、サービス改善 |

**試験での注意**: 開発・実装は「サービストランジション」、本番運用は「サービスオペレーション」に属します。

### ITIL v4 との関係

ITIL v4（2019年）では、**サービスバリューシステム（SVS）** という新しい枠組みが導入されました。v3 の 5 段階ではなく、「顧客価値を生み出す」という目的から逆算して設計します。ただし、試験では v3 の 5 段階が基準です。

## サービスオペレーション（本番運用の主要プロセス）

試験では「本番運用で何をするか」が問われるため、**サービスオペレーション** の代表的なプロセスを理解することが重要です。

### インシデント管理 vs 問題管理

この 2 つはしばしば混同されますが、**対象と目的が全く異なります**。

| 項目 | インシデント管理 | 問題管理 |
| --- | --- | --- |
| **定義** | サービス中断や品質低下の事象 | インシデントの根本原因 |
| **対象** | 症状（障害が起きている状態） | 原因（なぜ起きたのか） |
| **目的** | **素早く復旧させる** | **再発を防止する** |
| **時間軸** | 短期（すぐに対応） | 中期・長期（根本を解決） |
| **典型例** | データベースが落ちた → すぐに再起動 | DB が落ちる原因を調査 → メモリリークを発見 → パッチ適用 |
| **関連書類** | インシデントチケット | 問題記録、既知エラー DB |

**試験での見分け方**: 「とにかく直す」ならインシデント、「なぜそれが起きたか」なら問題。

### 変更管理（Change Management）

システムに加えるあらゆる変更（パッチ、設定変更、ソフトウェア更新など）を統制するプロセス。

- **目的**: 変更による予期しない障害を防ぐ
- **流れ**: 変更要求 → 評価・承認 → スケジューリング → 実施 → 確認
- **標準変更（Normal Change）**: ルーチン的な変更（自動化可）
- **緊急変更（Emergency Change）**: 障害復旧に必要な即座の変更（簡略プロセス）

### リリース管理（Release Management）

複数の変更を束ねて、一度に本番環境に導入するプロセス。

| 形態 | 説明 | 特徴 |
| --- | --- | --- |
| **フルリリース** | すべてのコンポーネントを同時に交換 | リスク大、テストしやすい |
| **段階的リリース** | 複数の場所に段階的に導入 | リスク分散、テスト期間が必要 |
| **パッチリリース** | 小さな修正を個別に導入 | 頻度多い、変更管理の負担大 |

### 構成管理（Configuration Management）

IT 環境を構成するあらゆる要素（ハードウェア、ソフトウェア、ライセンス、ドキュメント）の情報を一元管理します。

- **構成管理データベース（CMDB）**: 全資産の情報を記録
- **構成アイテム（CI）**: 管理対象（例：サーバー、OS、アプリケーション）
- **版管理**: どのバージョンが本番で動いているかを常に把握

### キャパシティ管理（Capacity Management）

システムの処理能力を計画的に管理し、パフォーマンスを保つプロセス。

- リソース（CPU、メモリ、ストレージ、ネットワーク）の使用状況を監視
- 将来の需要に備えて拡張を計画
- 費用とパフォーマンスのバランスを取る

### 可用性管理（Availability Management）

サービスが利用可能な状態をどれだけ保つかを管理します。

- **稼働率目標を設定** → **MTBF、MTTR の改善** → **冗長設計** などの施策
- 後述する MTBF、MTTR の改善と密接に関連

## SLA / SLO / OLA の 3 層構造

サービスレベルを管理する際、3 つのレベルがあります。試験では、この **3 者の関係** を理解することが重要です。

| 項目 | 対象 | 内容 | 典型例 |
| --- | --- | --- | --- |
| **SLA（Service Level Agreement）** | **利用者と提供者の合意** | 提供側が保証するサービスの水準 | 「稼働率 99.9%、応答時間 2 秒以内」 |
| **SLO（Service Level Objective）** | **内部目標（SLA の上位目標）** | 提供側が内部的に目指す基準 | SLA の 99.9% に対し、内部的に 99.95% を目指す |
| **OLA（Operational Level Agreement）** | **内部部門間の合意** | IT 部門内、または外部委託先との間の約束 | インフラ部門が「DB を 99.9% の稼働率で提供する」 |

**3 者の関係**: `OLA（内部の約束）` → `SLO（内部目標）` → `SLA（利用者への約束）`

図で表すと：
```
外部利用者との約束
    ↓
  【SLA】99.9% 稼働率
    ↓
【SLO】99.95% 稼働率（バッファ付き）
    ↓
【OLA】各部門（DB チーム、ネットワークチーム）が 99.9% を個別に達成
```

## サービスデスク（Service Desk）

利用者からの問い合わせ、要望、障害報告を受け付ける窓口です。

### 4 つの形態

| 形態 | 説明 | メリット | デメリット |
| --- | --- | --- | --- |
| **ローカルサービスデスク** | 各拠点に独立した窓口を設置 | 地域ニーズに対応しやすい | 運用費用が増加 |
| **中央サービスデスク** | 全利用者の問い合わせを一か所で処理 | コスト効率、知識の集約 | 地域対応に遅れ |
| **バーチャルサービスデスク** | メール、チャット、Web フォームなど複数チャネルを統合 | 24 時間対応可、低コスト | 初期構築コスト高 |
| **フォロー・ザ・サン** | 複数の地域サービスデスクが業務時間をカバー | 24 時間サポート、タイムゾーン対応 | 調整複雑、費用高 |

### SPOC（Single Point of Contact）

利用者にとって **唯一の窓口** を提供する原則です。

- 利用者は 1 つの番号、メールアドレスだけに連絡
- サービスデスクが内部で適切な部門に割り振る
- 利用者は複数部門との調整に気を配らない

## システム監査と内部統制

この 2 つは、「誰が」「どの立場で」チェックするかで分かれます。

### システム監査とは

**情報システムが有効かつ安全に運用されているかを、第三者的に点検する活動** です。

| 項目 | 説明 |
| --- | --- |
| **独立性** | 被監査部門から独立した立場で実施 |
| **客観性** | 証拠に基づいた評価 |
| **専門性** | IT と業務の知識が必要 |
| **継続性** | 定期的、または継続的に実施 |

### システム監査の流れ

```
① 予備調査
   ↓
② 本調査（詳細な調査）
   ↓
③ 評価・結論（監査意見の形成）
   ↓
④ 報告（監査報告書作成）
   ↓
⑤ フォローアップ（改善状況の確認）
```

### 監査証拠・監査調書・監査報告書

| 用語 | 意味 |
| --- | --- |
| **監査証拠** | 監査中に集めた事実の根拠（ログ、ドキュメント、インタビュー記録など） |
| **監査調書** | 監査証拠を整理した調査記録（監査人が保管） |
| **監査報告書** | 監査結果と意見を経営層に報告する文書（対外公開） |

### 助言型監査 vs 保証型監査

| 項目 | 助言型監査 | 保証型監査 |
| --- | --- | --- |
| **目的** | コンサルティング的に改善提案する | 既存の統制が有効かを客観的に評価 |
| **立場** | アドバイザー | 独立した評価者 |
| **典型例** | セキュリティ体制の構築に向けた提案 | 内部統制が機能しているかの確認 |

## 内部統制と J-SOX 法

### 内部統制とは

**業務や財務報告を適正に行うための、組織内の統制の仕組み全体** です。ITIL のような運用プロセスとは異なり、「法令遵守」「財務報告の信頼性」という経営層の責任に関わります。

### 4 つの目的（J-SOX 法に基づく）

| 目的 | 説明 | 例 |
| --- | --- | --- |
| **業務の有効性・効率性** | 目標達成に向けた効率的な運用 | 無駄のないプロセス設計、役割の明確化 |
| **財務報告の信頼性** | 会計情報の正確性・信頼性を確保 | 決算数字の正確性、帳簿と実績の一致 |
| **法令遵守** | 法律・規制に従う | 企業秘密管理、コンプライアンス |
| **資産保全** | 資産の流用・盗取を防ぐ | 在庫管理、アクセス制限 |

### 6 つの基本的要素

| 要素 | 説明 |
| --- | --- |
| **統制環境** | 経営層の姿勢、企業文化、倫理観が基盤 |
| **リスク評価** | リスクを識別し、優先度をつける |
| **統制活動** | 実際の統制施策（チェック、承認、分離など） |
| **情報と伝達** | 統制に必要な情報が、適切に関係者に伝わるか |
| **モニタリング** | 統制が継続的に機能しているか確認（監査など） |
| **IT への対応** | IT 全般統制と IT 業務処理統制 |

### IT 統制

内部統制のうち、情報システムに関わる部分。

#### IT 全般統制

システム全体を対象とした統制。

| 項目 | 説明 |
| --- | --- |
| **システム開発の統制** | 要件定義、設計、テスト、本番移行の各段階での品質確保 |
| **システム保守の統制** | パッチ管理、変更管理、バージョン管理 |
| **アクセス管理** | ユーザー認証、権限管理、ログ記録 |
| **バックアップ・復旧** | 定期的なバックアップ、災害復旧計画 |
| **組織・人員** | 責任分離（例：開発と運用の分離）、教育 |

#### IT 業務処理統制

個別の業務アプリケーションの統制。

| 項目 | 説明 |
| --- | --- |
| **入力統制** | データ入力の正確性チェック（形式、範囲、重複） |
| **処理統制** | 計算、ロジックの正確性確認 |
| **出力統制** | 出力データの検証、完全性確認 |

## システム監査 vs 内部統制の整理

試験ではこの 2 つを混同する受験生が多いため、はっきり分ける必要があります。

| 項目 | システム監査 | 内部統制 |
| --- | --- | --- |
| **立場** | 第三者的に点検 | 組織内部で日常的に実行 |
| **目的** | 有効性・安全性を評価する | 業務を適正に遂行する |
| **実施者** | 監査部門（独立） | 各部門の担当者 |
| **継続性** | 定期的または定点観測的 | 日常的・継続的 |
| **例** | 監査部が年 1 回、システムがちゃんと動いているか確認 | 毎日のアクセスログ記録、承認プロセスの実行 |

**図解**: 内部統制が「毎日の歯磨き」なら、システム監査は「年 1 回の歯科検診」

## 信頼性指標と稼働率計算

### RASIS モデル

IT システムの信頼性を 6 つの視点から評価します。**試験では、どの視点を問われているかを見分けることが重要**です。

| 指標 | 説明 | 評価方法 |
| --- | --- | --- |
| **R（Reliability）信頼性** | 故障しないか | MTBF（故障と故障の間隔） |
| **A（Availability）可用性** | 必要な時に使えるか | 稼働率 = MTBF / (MTBF + MTTR) |
| **S（Serviceability）保守性** | 修理しやすいか | MTTR（修理時間）、交換部品の入手性 |
| **I（Integrity）完全性** | データが壊れないか | エラー検出、バックアップ |
| **S（Security）安全性** | 不正アクセスから守られているか | アクセス制限、ログ記録 |

### MTBF と MTTR

稼働率計算の基礎となる 2 つの指標です。

#### MTBF（Mean Time Between Failures）

**「故障と故障の間の平均時間」**。故障しにくさを示します。

$$
\text{MTBF} = \frac{\text{稼働していた総時間}}{\text{故障の回数}}
$$

**例**: 1000 時間稼働中に 5 回故障した場合
$$
\text{MTBF} = \frac{1000}{5} = 200 \text{時間}
$$

#### MTTR（Mean Time To Repair）

**「故障から復旧までの平均時間」**。修理しやすさを示します。

$$
\text{MTTR} = \frac{\text{故障による停止時間の合計}}{\text{故障の回数}}
$$

**例**: 5 回の故障で合計 10 時間停止した場合
$$
\text{MTTR} = \frac{10}{5} = 2 \text{時間}
$$

### 稼働率（Availability）の計算

稼働率は、全体時間のうち「実際に使える時間」の割合です。

$$
\text{稼働率} = \frac{\text{MTBF}}{\text{MTBF} + \text{MTTR}}
$$

**例**: MTBF = 200 時間、MTTR = 2 時間の場合
$$
\text{稼働率} = \frac{200}{200 + 2} = \frac{200}{202} \approx 0.9901 \approx 99.01\%
$$

### 直列システムの稼働率

複数のコンポーネントが全て正常に動く必要がある場合（例：A → B → C の処理フロー）。

$$
\text{全体の稼働率} = R_1 \times R_2 \times R_3 \times \cdots \times R_n
$$

**例**: 3 つのサーバーが直列で接続
- サーバー A の稼働率: 0.9
- サーバー B の稼働率: 0.8
- サーバー C の稼働率: 0.95

$$
\text{全体の稼働率} = 0.9 \times 0.8 \times 0.95 = 0.684 = 68.4\%
$$

**ポイント**: どれか 1 つが落ちると全体が止まるため、稼働率は低くなる。

### 並列システムの稼働率

複数のコンポーネントがあり、1 つ以上が正常に動けば良い場合（例：冗長化されたデータベース）。

$$
\text{全体の稼働率} = 1 - (1 - R_1)(1 - R_2)(1 - R_3) \cdots (1 - R_n)
$$

**例**: 2 つのサーバーが並列冗長化
- サーバー 1 の稼働率: 0.9
- サーバー 2 の稼働率: 0.9

$$
\text{全体の稼働率} = 1 - (1 - 0.9)(1 - 0.9) = 1 - 0.1 \times 0.1 = 1 - 0.01 = 0.99 = 99\%
$$

**ポイント**: 1 つが落ちても他が動くため、稼働率は向上する。

### 複合システムの稼働率（直列と並列の組み合わせ）

実際のシステムは直列と並列が混在することが多いです。

**例**: 次の構成
```
Web サーバー（1 台、稼働率 0.95）
    ↓
DB サーバー（2 台並列冗長、それぞれ稼働率 0.9）
    ↓
ストレージ（3 台並列冗長、それぞれ稼働率 0.92）
```

**Step 1**: DB 2 台の並列稼働率
$$
R_{DB} = 1 - (1 - 0.9)^2 = 1 - 0.01 = 0.99
$$

**Step 2**: ストレージ 3 台の並列稼働率
$$
R_{Storage} = 1 - (1 - 0.92)^3 = 1 - 0.08^3 = 1 - 0.000512 \approx 0.9995
$$

**Step 3**: Web → DB → Storage が直列
$$
\text{全体の稼働率} = 0.95 \times 0.99 \times 0.9995 \approx 0.9405 \approx 94.05\%
$$

## フォールトトレランス設計（Fault Tolerance）

システムが故障した場合、どのように対応するかという設計方針です。試験では 3 つの基本的な考え方と、その実装方式（デュプレックス、デュアルなど）を理解する必要があります。

### 3 つの基本原則

| 原則 | 説明 | 典型例 |
| --- | --- | --- |
| **フェールセーフ（Fail-Safe）** | 故障時に、安全な状態に移る | 信号機が落ちたら全て赤 → 事故を防ぐ |
| **フェールソフト（Fail-Soft）** | 故障時に、機能を制限して運用を続ける | 1 台が落ちても他で処理を継続 |
| **フールプルーフ（Fool-Proof）** | 誤操作を物理的に防ぐ | USB の向きが 1 方向だけ→逆さまに挿せない |

**試験での見分け方**:
- 「安全第一」 → フェールセーフ
- 「止めずに続ける」 → フェールソフト
- 「誤操作防止」 → フールプルーフ

### 冗長化システムの形態

| 形態 | 説明 | 復旧時間 | 費用 | 利用例 |
| --- | --- | --- | --- | --- |
| **ホットスタンバイ（Hot Standby）** | 待機系が常に稼働、同期。障害時に即座に切り替え | 数秒以内 | 高い | ミッションクリティカルなシステム |
| **コールドスタンバイ（Cold Standby）** | 待機系が停止している。障害時に起動して復旧 | 数時間 | 低い | バッチ処理、事務系システム |
| **ウォームスタンバイ（Warm Standby）** | 待機系がデータを定期的に同期。中間的な復旧時間 | 10～30 分 | 中程度 | 一般的なオンラインシステム |

### デュプレックスシステム

2 つの同一システムを並列配置します。

```
     入力
      ↓
  ┌────────┐
  │ 合意器 │  ← 2 台の結果が一致しているか確認
  └────────┘
    ↓   ↓
┌─────────────┐
│ System A    │
│(Active)     │
└─────────────┘
┌─────────────┐
│ System B    │
│(Standby)    │
└─────────────┘
```

- **機能**: 2 台が常に同じ状態を保ち、片方が故障しても他方で継続
- **フェールセーフ機能**: 両者の結果が異なったら安全な状態に

### デュアルシステム

2 つの独立したシステムが、同時に同じ処理を行います。結果の合致を確認してから出力。

```
     入力
      ↓
  ┌─────────────────┐
  │   System A      │
  └─────────────────┘
            ↓
       ┌────────┐
       │ 合意器 │  ← 結果の一致を確認
       └────────┘
            ↓
  ┌─────────────────┐
  │   System B      │
  └─────────────────┘
      ↓
    出力
```

- **特徴**: 両者の結果が異なった場合、どちらも出力しない（安全重視）
- **用途**: 航空機、医療機器など、安全が最優先の場面

### デュプレックス vs デュアルの比較

| 項目 | デュプレックス | デュアル |
| --- | --- | --- |
| **構成** | Active-Standby（1 つが稼働） | 両者が同時稼働 |
| **故障時** | Standby に即座に切り替え | 両者を停止 |
| **特性** | 高い稼働率、可用性重視 | 高い信頼性、安全性重視 |
| **用途** | Webサービス、DBサーバー | 航空機、原発制御 |

## 典型的なつまずき

### よくある誤解と解説

1. **「ITIL はツール名」と思う**
   - ❌ ITIL は製品ではなく、ベストプラクティス集
   - ✅ ITIL の考え方に沿ったツール（ServiceNow など）がある

2. **「SLA が全ての運用目標」と混同する**
   - ❌ SLA は「利用者との合意」だけ
   - ✅ 内部目標（SLO）、部門間の約束（OLA）も同時に管理

3. **「インシデントと問題は同じ」と見なす**
   - ❌ 症状と原因を混同
   - ✅ 症状（インシデント）を素早く直し、原因（問題）を根本解決する並行活動

4. **「MTBF が高い = 稼働率が高い」と誤解**
   - ❌ 計算が必要
   - ✅ `稼働率 = MTBF / (MTBF + MTTR)` で計算。MTTR が大きければ稼働率は低い

5. **「システム監査と内部統制は両方チェック」と同視**
   - ❌ 立場が違う
   - ✅ 内部統制が「毎日の活動」なら、監査は「定期的な検証」

6. **「並列システムは単純に稼働率を足す」と計算**
   - ❌ 足し算ではない
   - ✅ `1 - (1-R1)(1-R2)` という乗算で計算

7. **「フェールセーフ = 止まる」と誤解**
   - ❌ 「安全な状態」は必ずしも「停止」ではない
   - ✅ 信号機の例では赤信号（安全状態）で運用継続

## 問題を解くときの観点

試験問題を読む際は、以下の順序で考えます。

### ステップ 1: 「何のプロセス/指標か」を見分ける

| 問われている内容 | 焦点 | 該当項目 |
| --- | --- | --- |
| 「開発後のシステムを安定稼働させる」「品質を保つ」 | 運用管理 | ITIL のプロセス |
| 「利用者との合意水準」「目標値」 | 合意 | SLA / SLO / OLA |
| 「サービス品質の評価・指標」 | 評価 | 稼働率、MTBF、MTTR |
| 「第三者的に確認」「チェック」 | 点検 | システム監査 |
| 「日常的に統制する仕組み」 | 統制 | 内部統制 |

### ステップ 2: 「何が対象か」を確認

| 言葉 | 意味 | 対応項目 |
| --- | --- | --- |
| 「障害が発生した」「サービスが停止した」 | 事象（症状） | インシデント |
| 「なぜ障害が発生したのか」「根本原因」 | 原因 | 問題管理 |
| 「本番環境に導入する」「バージョン更新」 | デプロイメント | リリース管理、変更管理 |
| 「全資産を記録する」「バージョン管理」 | 資産管理 | 構成管理 |

### ステップ 3: 「時間軸は短期か長期か」で判定

| 時間軸 | 対応項目 | 目的 |
| --- | --- | --- |
| **数分～数時間（短期）** | インシデント対応、フェールオーバー | 即座に復旧 |
| **数日～数週間（中期）** | 問題管理、変更管理 | 根本解決、再発防止 |
| **年単位（長期）** | キャパシティ管理、内部統制 | 計画的な改善 |

### ステップ 4: 計算問題は単位を確認

| 単位 | 計算方法 | 注意点 |
| --- | --- | --- |
| **MTBF（時間）** | 稼働時間 ÷ 故障回数 | 故障しにくさ |
| **MTTR（時間）** | 停止時間 ÷ 故障回数 | 修理しやすさ |
| **稼働率（% または小数）** | MTBF / (MTBF + MTTR) | 0～1 または 0～100% |
| **直列の稼働率** | R1 × R2 × ... | かけ算 |
| **並列の稼働率** | 1 - (1-R1)(1-R2)... | 1 から引く |

## 確認問題

<Callout title="問1：インシデント管理と問題管理の違い">
    **問題文**

    あるオンラインバンキングシステムで、突然データベースが応答しなくなった。利用者がアクセスできない状態になった。これに対し、以下の 2 つの活動が行われた。

    - A: すぐにデータベースを再起動し、サービスを復旧させた
    - B: 再起動後、メモリ使用量のログを分析し、メモリリークが原因であることを発見。アプリケーションのパッチを適用した

    A と B の活動をそれぞれ何管理と呼ぶか。また、両者の違いを簡潔に説明せよ。

    **解答**

    - **A は「インシデント管理」**
    - **B は「問題管理」**

    **違い**:
    - インシデント管理（A）: サービスが停止している「症状」に対し、**素早く復旧させることが目的**。原因は後回し。
    - 問題管理（B）: 復旧後、「なぜこれが起きたのか」という**根本原因を調査**し、**パッチなどで再発を防止するのが目的**。

    **試験での見分け方**: 「とにかく直す」はインシデント、「なぜそれが起きたのか」は問題。
</Callout>

<Callout title="問2：直列システムと並列システムの稼働率計算">
    **問題文**

    あるシステムが以下の構成になっている。

    ```
    Web サーバー（稼働率 0.92）
           ↓
    App サーバー（稼働率 0.95）
           ↓
    DB サーバー（稼働率 0.90）
    ```

    この 3 台が直列で接続されている場合、全体の稼働率を計算せよ。

    さらに、DB サーバーを 2 台の並列冗長に変更した場合（各稼働率 0.90）、新しい全体稼働率を計算せよ。

    **解答**

    **① 直列の場合**:
    $$
    \text{全体の稼働率} = 0.92 \times 0.95 \times 0.90 = 0.7866 \approx 78.66\%
    $$

    **② DB を 2 台並列にした場合**:

    まず、DB 2 台の並列稼働率を計算:
    $$
    R_{DB} = 1 - (1 - 0.90)^2 = 1 - 0.10^2 = 1 - 0.01 = 0.99
    $$

    次に、Web → App → DB（並列）が直列:
    $$
    \text{全体の稼働率} = 0.92 \times 0.95 \times 0.99 = 0.8646 \approx 86.46\%
    $$

    **結論**: DB を冗長化することで、78.66% → 86.46% に改善。
</Callout>

<Callout title="問3：SLA、SLO、OLA の関係と役割">
    **問題文**

    ある企業が SaaS 型の顧客管理システムを導入した。以下の 3 つの合意がある。

    - 利用部門と IT 部門の間では「稼働率 99.5%」という約束
    - IT 部門が顧客に対して「稼働率 99.9%」を保証
    - インフラ部門は IT 部門内で「稼働率 99.8%」を目指す

    これら 3 つの関係を、SLA、SLO、OLA のいずれかに分類し、それぞれの役割を説明せよ。

    **解答**

    | 合意内容 | 該当 | 役割 | 説明 |
    | --- | --- | --- | --- |
    | 「利用部門と IT 部門の間で稼働率 99.5%」 | **OLA** | 部門間の合意 | IT 部門内（インフラチーム）が、他部門（利用部門）に対して果たすべき約束 |
    | 「顧客に稼働率 99.9% を保証」 | **SLA** | 利用者（外部）との合意 | 企業が顧客に対して公式に約束した水準。契約で定められることが多い |
    | 「インフラ部門が稼働率 99.8% を目指す」 | **SLO** | 内部目標 | SLA（99.9%）を確実に達成するため、内部的により高い目標（99.8%）を設定 |

    **図式**:
    ```
    顧客への約束（SLA: 99.9%）
           ↓
    内部目標（SLO: 99.8%）  ← バッファを持たせる
           ↓
    部門間合意（OLA: 99.5%） ← インフラチームがエンジニアリング部門に約束
    ```

    **試験での重要性**: 外部（SLA）と内部（SLO、OLA）を明確に分けることで、過度な目標設定を避け、実現可能な品質管理ができる。
</Callout>
