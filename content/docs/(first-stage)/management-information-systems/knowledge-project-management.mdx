## PMBOK の全体構造

### PMBOK 5つのプロセス群

プロジェクトマネジメントは、時系列で5つのプロセス群に分かれます。

| プロセス群 | タイミング | 主な活動 |
| --- | --- | --- |
| **立上げ（Initiating）** | プロジェクト開始前 | 目的・スコープの確定、ステークホルダー特定 |
| **計画（Planning）** | 立上げ直後 | WBS、スケジュール、予算、品質基準の計画 |
| **実行（Executing）** | 作業期間中 | チーム展開、成果物作成、コミュニケーション |
| **監視・コントロール（Monitoring & Controlling）** | 作業期間中・並行 | 進捗確認、偏差分析、予防処置・是正処置 |
| **終結（Closing）** | プロジェクト終了時 | 成果物受け入れ、教訓整理、契約終了 |

### PMBOK 10の知識エリア

各プロセス群を支える10個の管理領域があります。試験では、「どの領域でどのプロセスが関わるか」を理解することが重要です。

| 知識エリア | 主な対象 | 典型的な活動 |
| --- | --- | --- |
| **統合管理（Integration）** | 全体 | 計画統合、変更管理、プロジェクトチャーター作成 |
| **スコープ管理（Scope）** | 計画・監視 | WBS作成、スコープ確定、スコープ変更管理 |
| **スケジュール管理（Schedule）** | 計画・監視 | アクティビティ定義、PERT/CPM、日程表作成 |
| **コスト管理（Cost）** | 計画・監視 | 見積り、予算化、EVM による進捗管理 |
| **品質管理（Quality）** | 計画・実行・監視 | 品質基準設定、品質保証、検査・検証 |
| **資源管理（Resource）** | 計画・実行 | チーム配置、スキル管理、リソース調整 |
| **コミュニケーション管理（Communications）** | 計画・実行・監視 | 情報配信計画、会議、報告書 |
| **リスク管理（Risk）** | 計画・監視 | リスク特定、分析、対応計画、監視 |
| **調達管理（Procurement）** | 計画・実行・終結 | 外注判定、入札、契約管理 |
| **ステークホルダー管理（Stakeholder）** | 全体 | 関係者分析、期待値管理、参加促進 |

## WBS（作業分解構造）

### 定義と目的

**WBS（Work Breakdown Structure）** は、プロジェクトのスコープを階層的に小さな作業（ワークパッケージ）に分解する道具です。

- **目的**：スコープの見える化、担当・期間・予算の把握漏れ防止、責任の明確化
- **構造**：最上位（プロジェクト全体）→ 中間（フェーズ・カテゴリ）→ 最下位（ワークパッケージ）
- **成果物**：WBS ツリー図、WBS ディクショナリ（各項目の説明と責任者）

### WBS の例（システム開発プロジェクト）

```
プロジェクト：新販売管理システム
├─ 1. 計画フェーズ
│  ├─ 1.1 要件定義
│  ├─ 1.2 基本設計
│  └─ 1.3 計画文書作成
├─ 2. 開発フェーズ
│  ├─ 2.1 詳細設計
│  ├─ 2.2 実装（モジュールA, B, C）
│  └─ 2.3 単体テスト
├─ 3. テスト・導入フェーズ
│  ├─ 3.1 統合テスト
│  ├─ 3.2 ユーザー受入テスト
│  └─ 3.3 本番導入
└─ 4. クローズアウト
   ├─ 4.1 移行支援
   └─ 4.2 教訓整理
```

### WBS と他の計画手法の関係

| 手法 | 対象 | 用途 | 違い |
| --- | --- | --- | --- |
| **WBS** | スコープ | **何を作るか**の分解 | 階層分解、要素の洗い出し |
| **PERT/CPM** | スケジュール | **どの順序で、いつまでか** | 作業の順序関係と日程 |
| **EVM** | 進捗・コスト | **どこまで進んだか、いくら使ったか** | 進捗とコストの同時評価 |

## PERT と CPM

### アローダイアグラムの基本

**アローダイアグラム** は、作業（アクティビティ）を矢線で表し、順序関係を図示する方法です。

**基本要素**：
- **節点（ノード）**：作業の開始と終了を表す円または四角
- **矢線（アロー）**：作業を表す矢線。矢線の長さは時間を表さない
- **ダミー矢線**：実際の作業ではなく、作業間の順序関係を表すために引く点線

### アローダイアグラムの描き方ルール

1. **先行作業のない作業は、開始節点から出発**
2. **後続作業のない作業は、終了節点に到達**
3. **同じ節点から出る矢線と、その節点に入る矢線の意味は異なる**
4. **順序を表すためにダミー矢線を使う場合がある**

### 最早開始時刻（ES）と最早完了時刻（EF）の計算

**前向き計算**：開始節点から終了節点へ進める

- `ES（最早開始時刻）` = その作業に到達する直前の作業の EF の最大値
- `EF（最早完了時刻）` = `ES + 作業期間`

### 最遅開始時刻（LS）と最遅完了時刻（LF）の計算

**後ろ向き計算**：終了節点から開始節点へ遡る

- `LF（最遅完了時刻）` = その作業から出発する直後の作業の LS の最小値
- `LS（最遅開始時刻）` = `LF - 作業期間`

### トータルフロート（総余裕時間）

作業に遅れが生じても全体日程に影響しない余裕時間を **トータルフロート** といいます。

$$\text{トータルフロート} = LS - ES = LF - EF$$

- **フロート = 0**：その作業は**クリティカルパス上**にあり、遅れると全体が遅れる
- **フロート > 0**：その作業には余裕があり、遅れても大丈夫（最大でもフロート分まで）

### 具体的な計算例

次のネットワーク図を考えます。

**作業リスト**：

| 作業 | 先行作業 | 所要期間（日） |
| --- | --- | --- |
| A | なし | 5 |
| B | A | 3 |
| C | A | 4 |
| D | B | 2 |
| E | C | 6 |
| F | D, E | 3 |

**前向き計算（ES, EF）**：

```
A:  ES=0,   EF=0+5=5
B:  ES=5,   EF=5+3=8
C:  ES=5,   EF=5+4=9
D:  ES=8,   EF=8+2=10
E:  ES=9,   EF=9+6=15
F:  ES=max(10, 15)=15,   EF=15+3=18
```

プロジェクト全体の完了予定：**18日**

**後ろ向き計算（LF, LS）**：

プロジェクト全体の完了時刻 LF(終了) = 18 から開始

```
F:  LF=18,   LS=18-3=15
D:  LF=15,   LS=15-2=13
E:  LF=15,   LS=15-6=9
B:  LF=13,   LS=13-3=10
C:  LF=9,    LS=9-4=5
A:  LF=min(10, 5)=5,   LS=5-5=0
```

**トータルフロートの計算**：

| 作業 | ES | LS | フロート | 状態 |
| --- | --- | --- | --- | --- |
| A | 0 | 0 | 0 | **クリティカルパス** |
| B | 5 | 10 | 5 | 余裕あり |
| C | 5 | 5 | 0 | **クリティカルパス** |
| D | 8 | 13 | 5 | 余裕あり |
| E | 9 | 9 | 0 | **クリティカルパス** |
| F | 15 | 15 | 0 | **クリティカルパス** |

**クリティカルパス**：A → C → E → F（18日）

- これらの作業が 1 日でも遅れると、プロジェクト全体が遅れる
- 作業 B、D には余裕があるため、最大 5 日まで遅れても全体には影響しない

## EVM（獲得価値法）

### EVM の3つの基本変数

**EVM** は、計画と実績をお金の尺度で比較し、進捗とコスト効率を同時に管理する手法です。

| 変数 | 定義 | 計算方法 |
| --- | --- | --- |
| **PV（計画価値）** | 今の時点で「予定通りなら完了しているはずの成果に対応する予算」 | 当初予算（BAC）× 計画進捗率 |
| **EV（獲得価値）** | 「現在まで実際に完了した成果に対応する予算」 | 当初予算（BAC）× 実績進捗率 |
| **AC（実際原価）** | 「現在までに実際に使った金額」 | 実績支出額 |

### EVM から導出される5つの指標

#### 1. スケジュール差異（SV）

$$SV = EV - PV$$

- **SV > 0**：進捗が予定より**早い**
- **SV < 0**：進捗が予定より**遅い**
- **SV = 0**：予定通り

#### 2. コスト差異（CV）

$$CV = EV - AC$$

- **CV > 0**：予算より**少ない**支出（効率的）
- **CV < 0**：予算より**多い**支出（非効率）
- **CV = 0**：予定通り

#### 3. スケジュール効率指数（SPI）

$$SPI = \frac{EV}{PV}$$

- **SPI > 1**：進捗が予定より**早い**
- **SPI < 1**：進捗が予定より**遅い**
- **SPI = 1**：予定通り

#### 4. コスト効率指数（CPI）

$$CPI = \frac{EV}{AC}$$

- **CPI > 1**：1円の支出で予定より多くの価値（効率的）
- **CPI < 1**：1円の支出で予定より少ない価値（非効率）
- **CPI = 1**：予定通り

#### 5. 完成時総コスト見積り（EAC）

$$EAC = \frac{BAC}{CPI}$$

- プロジェクト全体の最終予想支出額
- **EAC > BAC**：予算超過の見込み
- **EAC < BAC**：予算内で完了の見込み

### EVM の計算例

**プロジェクト概要**：
- 当初予算（BAC）：1000万円
- 現在時点の計画進捗率：50%（中間地点）
- 現在時点の実績進捗率：45%
- 現在までの実支出（AC）：420万円

**ステップ1：基本変数を計算**

```
PV = BAC × 計画進捗率 = 1000万円 × 50% = 500万円
EV = BAC × 実績進捗率 = 1000万円 × 45% = 450万円
AC = 420万円
```

**ステップ2：差異を計算**

```
SV = EV - PV = 450万円 - 500万円 = -50万円
→ 進捗が予定より50万円分遅れている（スケジュール遅延）

CV = EV - AC = 450万円 - 420万円 = 30万円
→ 実績が予定より30万円分効率的（コスト効率が良い）
```

**ステップ3：効率指数を計算**

```
SPI = EV / PV = 450万円 / 500万円 = 0.90
→ 計画の90%の進度で進行中（遅い）

CPI = EV / AC = 450万円 / 420万円 = 1.071
→ 1万円の支出で1.071万円分の価値が生まれている（効率的）
```

**ステップ4：完成時総コスト見積りを計算**

```
EAC = BAC / CPI = 1000万円 / 1.071 = 934万円
→ 最終的には約934万円で完成する見込み（予算内で完了）
```

### EVM の判定マトリクス

| SV の符号 | CV の符号 | 状況 | 対策 |
| --- | --- | --- | --- |
| **負**（遅い） | **負**（超過） | **最悪** | スケジュール短縮・リソース追加 |
| **負**（遅い） | **正**（効率的） | スケジュール遅延 | 優先順位変更・並行化 |
| **正**（早い） | **負**（超過） | コスト超過 | 支出抑制・効率化 |
| **正**（早い） | **正**（効率的） | **最良** | 現状維持 |

## 見積り技法

### 1. ファンクションポイント法（FP法）

**定義**：ソフトウェアの機能を5つのタイプに分類し、複雑度を加味して規模を算出する方法

**5つの機能タイプ**：

| 機能タイプ | 定義 | 複雑度 | 調整係数 |
| --- | --- | --- | --- |
| **入力（Input）** | ユーザーがシステムへ入力するデータ | 低・中・高 | 3～6 |
| **出力（Output）** | システムがユーザーへ出力するデータ | 低・中・高 | 4～7 |
| **照会（Inquiry）** | ユーザーが照会・検索するデータ | 低・中・高 | 3～6 |
| **内部ファイル（Internal File）** | システムが保有するデータベース | 低・中・高 | 7～10 |
| **外部ファイル（External File）** | 外部システムが保有し、本システムが参照するデータ | 低・中・高 | 5～7 |

**計算例**：

```
入力：5項目（複雑度：中平均） = 5 × 4 = 20FP
出力：3項目（複雑度：高） = 3 × 7 = 21FP
照会：4項目（複雑度：低） = 4 × 3 = 12FP
内部ファイル：2個（複雑度：中） = 2 × 8 = 16FP
外部ファイル：1個（複雑度：低） = 1 × 5 = 5FP
─────────────────────────
総ファンクションポイント = 74FP

規模見積り：開発生産性が1FPあたり10人日の場合
開発工数 = 74FP × 10人日/FP = 740人日
```

### 2. COCOMO（構成的コスト見積りモデル）

**定義**：開発規模（KLOC：キロ行）に基づいて、開発工数と期間を見積もる方法

**開発規模の3つのモード**：

| モード | 開発規模（KLOC） | 特徴 | 計算式 |
| --- | --- | --- | --- |
| **基本モード** | 30KLOC未満 | 小規模、シンプル | 工数(人月) = 2.4 × (KLOC)^1.05 |
| **中間モード** | 30～300KLOC | 中規模、複雑度中 | 工数(人月) = 3.0 × (KLOC)^1.12 |
| **詳細モード** | 300KLOC以上 | 大規模、複雑 | 工数(人月) = 3.6 × (KLOC)^1.20 |

**計算例**（基本モード、50KLOCの場合）：

```
工数 = 2.4 × (50)^1.05 = 2.4 × 56.4 = 135.4人月
開発期間（月） = 2.5 × (工数)^0.38 = 2.5 × (135.4)^0.38 ≈ 13.2ヶ月
```

### 3. 三点見積り

**定義**：楽観値、最可能値、悲観値の3つの推定値から、期待値を計算する方法

**計算式**（ベータ分布を仮定）：

$$\text{期待値} = \frac{\text{楽観値} + 4 \times \text{最可能値} + \text{悲観値}}{6}$$

**計算例**：

```
作業A：
  楽観値（最良の場合）：5日
  最可能値（最も起こりやすい）：8日
  悲観値（最悪の場合）：14日

期待値 = (5 + 4×8 + 14) / 6 = (5 + 32 + 14) / 6 = 51 / 6 = 8.5日
```

### 4. 類推見積り

**定義**：過去の類似プロジェクトの実績を基に、新規プロジェクトの工数を見積もる方法

**利点**：
- 過去データを活用するため、信頼度が高い
- 組織固有の生産性を反映できる

**課題**：
- 類似プロジェクトが存在しない場合は使えない
- 類似度の判定が主観的になりやすい

### 5. ボトムアップ見積り

**定義**：プロジェクトを WBS まで分解し、最下位のワークパッケージごとに見積もって、積み上げる方法

**プロセス**：

```
総工数 = Σ(各ワークパッケージの工数)
```

**利点**：
- 詳細な作業を起点にするため、精度が高い
- 各担当者の意見を反映できる

**課題**：
- 分解と見積りに時間がかかる
- WBS が不完全だと漏れが生じる

## リスクマネジメント

### リスク管理の5ステップ

プロジェクトリスク（プロジェクトが失敗する要因）を組織的に管理するプロセス：

| ステップ | 活動 | 出力 |
| --- | --- | --- |
| **1. リスク特定** | 起こりうるリスクを洗い出す | リスク登録簿 |
| **2. 定性的分析** | リスクの発生可能性と影響度で分類 | リスク優先順位 |
| **3. 定量的分析** | リスクの金銭的インパクトを計算 | 期待値（確率 × 金額） |
| **4. リスク対応計画** | 各リスクへの対応策を決定 | リスク対応計画書 |
| **5. リスク監視・制御** | リスク発生状況を監視、対応を実行 | リスク監視報告 |

### リスク評価マトリクス（定性的分析）

リスクを「発生確率」と「影響度」で 2×2 または 3×3 マトリクスに配置し、優先順位を判定

```
        影響度
        高 |
           |    中    高
        中 |  低     中
           | 低     中     高
        低 |_____________
             低    中     高  →発生確率
```

- **高優先度（赤）**：発生確率が高く、影響度が大きいリスク → 積極的対応
- **中優先度（黄）**：中程度のリスク → 監視と部分的対応
- **低優先度（青）**：軽微なリスク → 受容

### マイナスのリスク対応4戦略

起こると困るリスク（脅威）への対応：

| 戦略 | 説明 | 例 |
| --- | --- | --- |
| **回避（Avoid）** | リスクが発生しないように、プロジェクト方針・計画を変更 | 新しいツール導入を避け、既知のツールを使う |
| **転嫁（Transfer）** | リスクの影響を第三者に移す | 外注化、保険、外部委託 |
| **軽減（Mitigate）** | リスク発生の確率または影響度を低減 | テスト期間を延長、バックアップ体制構築 |
| **受容（Accept）** | リスク発生時の対応策を準備し、発生を受け入れる | 予備費を確保、緊急対応チーム整備 |

### プラスのリスク対応4戦略

起こるとよいリスク（機会）への対応：

| 戦略 | 説明 | 例 |
| --- | --- | --- |
| **活用（Exploit）** | 機会が確実に実現されるよう、プロジェクト計画を変更 | 優秀なメンバーの配置を確保 |
| **共有（Share）** | 機会の利益を第三者と共有する | ベンダーとの協業、パートナーシップ |
| **強化（Enhance）** | 機会の発生確率や利益を増大させる | 追加リソース投入、早期実施 |
| **受容（Accept）** | 機会を活かさず、現在の計画を維持 | 「現在の予算・スケジュール内で十分」と判定 |

### 期待値による定量的評価

リスクの金銭的インパクト：

$$\text{期待値} = \text{発生確率} \times \text{影響度（金額）}$$

**例**：
- リスク A：確率 30%、影響額 500万円 → 期待値 = 0.3 × 500 = 150万円
- リスク B：確率 70%、影響額 200万円 → 期待値 = 0.7 × 200 = 140万円
- リスク C：確率 10%、影響額 1000万円 → 期待値 = 0.1 × 1000 = 100万円

期待値が大きいリスク（リスク A）を優先的に対応します。

### リスク対応計画の例

| リスク | 発生確率 | 影響度 | 対応戦略 | 具体的対応 | 責任者 |
| --- | --- | --- | --- | --- | --- |
| 要件変更の頻発 | 高（60%） | 大 | 軽減 | 変更管理プロセス強化、要件レビューの徹底 | PM |
| キー人物の退職 | 中（20%） | 大 | 軽減 | ナレッジ共有化、バックアップ人員育成 | HR |
| ベンダー遅延 | 中（40%） | 中 | 転嫁 | SLA（サービスレベル契約）締結、複数ベンダー検討 | 調達 |
| ユーザー受け入れ遅延 | 低（15%） | 大 | 受容 | 予備期間を20%確保 | PM |

## 典型的なつまずき

1. **PMBOK の10知識エリアと5プロセス群の対応を混同**
   - 誤り：「スケジュール管理は計画プロセスだけで行われる」
   - 正解：計画フェーズで計画し、監視・コントロールフェーズで実績と比較

2. **WBS を日程表やガントチャートと勘違い**
   - 誤り：WBS に時系列順序や開始・終了日を記入する
   - 正解：WBS は「何を作るか」の階層分解。時系列は別のツール（スケジュール表）

3. **クリティカルパスを「作業が多い経路」だと誤解**
   - 誤り：「作業 A→B→C→D が最も多いから、クリティカルパス」
   - 正解：**最長経路（所要時間が最も長い経路）** がクリティカルパス。短い作業でも総時間が長ければクリティカル

4. **トータルフロートと自由フロートを区別しない**
   - 誤り：「作業 B のフロートは5日」だけで対応策を決定
   - 正解：トータルフロートは「自分の遅れが後続作業に影響しない最大日数」。後続作業がある場合は自由フロートで判定

5. **EVM の SV と CV を同じ概念と見なす**
   - 誤り：「SV = -50万円だから、コストが50万円超過している」
   - 正解：SV はスケジュール差異（時間軸）、CV はコスト差異（金額軸）。別の指標

6. **EVM で「進捗率」と「獲得価値」を混同**
   - 誤り：「実績進捗率が45%なら、EV = 450万円」（BAC=1000万円と仮定）
   - 正解：進捗率と EV は、一見同じに見えるが、計算の根拠が異なる場合がある。実績成果物をしっかり定義してから EV を算出

7. **見積り手法を「どれが最も正確か」で比較**
   - 誤り：「COCOMO は信頼度が低い」と決めつける
   - 正解：プロジェクトの段階と情報量で選択。初期段階は類推、詳細化後はボトムアップが効果的

8. **リスク対応4戦略をすべてのリスクに適用しようとする**
   - 誤り：「すべてのリスクを回避する」
   - 正解：回避できるリスク、転嫁できるリスク、軽減するべきリスク、受容すべきリスクを分類

9. **リスク監視を怠り、実行中の新しいリスクに気づかない**
   - 誤り：「計画段階でリスク対応を決めたから、後は実行するだけ」
   - 正解：プロジェクト進行中に新規リスクが発見される。定期的にリスク登録簿を更新、監視

10. **変更管理と リスク対応を分離**
    - 誤り：「要件変更は変更管理、リスク対応は別」
    - 正解：変更は新たなリスク（スケジュール・コスト超過の原因）。変更管理とリスク管理は統合的に

---

## 問題を解くときの観点

1. **問題文から「何の段階か」を読む**
   - 「計画段階」か「実行・監視段階」か「終結段階」か
   - PMBOK の5つのプロセス群のどこにあるか

2. **「何を管理するのか」を特定**
   - スコープ（WBS）か、スケジュール（PERT）か、コスト（見積り・EVM）か、リスクか
   - 10の知識エリアのどれに該当するか

3. **数値が与えられた場合は、どの計算公式を使うか**
   - PERT：ES、EF、LS、LF、フロート
   - EVM：SV、CV、SPI、CPI、EAC
   - 見積り：FP法、COCOMO、三点見積り

4. **複数の手法が出てきた場合、使い分けを判定**
   - WBS と PERT：分解と日程管理は別
   - PERT と EVM：日程と進捗・コストは別
   - 見積り技法：段階と情報量で選択

5. **リスク問題では対応戦略の区別**
   - マイナスのリスク（脅威）：回避、転嫁、軽減、受容
   - プラスのリスク（機会）：活用、共有、強化、受容
   - 起こる前か、起きた後か

6. **問題解きのチェックリスト**
   - 数値計算は、公式に正しく値を代入したか
   - トータルフロート = 0 の作業がクリティカルパスか
   - EAC は BAC より大きいか小さいか（超過か削減か）を確認
   - リスク対応は、確率と影響度のバランスで決断しているか

## 確認問題

<Callout title="問1：アローダイアグラムとクリティカルパス">

次の作業リストからアローダイアグラムを描き、クリティカルパスと各作業のトータルフロートを求めてください。

| 作業 | 先行作業 | 所要期間（日） |
| --- | --- | --- |
| A | なし | 4 |
| B | A | 3 |
| C | A | 5 |
| D | B, C | 2 |

**解答**：

前向き計算（ES, EF）：
```
A:  ES=0,   EF=4
B:  ES=4,   EF=7
C:  ES=4,   EF=9
D:  ES=max(7,9)=9,   EF=11
```

プロジェクト完了：11日

後ろ向き計算（LF, LS）：
```
D:  LF=11,   LS=9
B:  LF=9,    LS=6
C:  LF=9,    LS=4
A:  LF=min(6,4)=4,   LS=0
```

トータルフロート：
- A：LS - ES = 0 - 0 = 0（**クリティカルパス**）
- B：LS - ES = 6 - 4 = 2（余裕2日）
- C：LS - ES = 4 - 4 = 0（**クリティカルパス**）
- D：LS - ES = 9 - 9 = 0（**クリティカルパス**）

**クリティカルパス：A → C → D（11日）**

作業 B だけに 2 日間の余裕があり、最大 2 日まで遅れても全体工程に影響しません。

</Callout>

<Callout title="問2：EVM による進捗・コスト分析">

次のデータから、SV、CV、SPI、CPI、EAC を計算し、プロジェクトの健全性を評価してください。

- 当初予算（BAC）：2000万円
- プロジェクト進捗率：計画 60%、実績 55%
- 現在までの実支出（AC）：1100万円

**解答**：

ステップ1：基本変数
```
PV = BAC × 計画進捗率 = 2000万円 × 60% = 1200万円
EV = BAC × 実績進捗率 = 2000万円 × 55% = 1100万円
AC = 1100万円
```

ステップ2：差異
```
SV = EV - PV = 1100万円 - 1200万円 = -100万円
→ スケジュール遅延：予定より100万円分遅れている

CV = EV - AC = 1100万円 - 1100万円 = 0万円
→ コスト効率：予定通り
```

ステップ3：効率指数
```
SPI = EV / PV = 1100 / 1200 = 0.917
→ 計画の91.7%のペースで進行（遅い）

CPI = EV / AC = 1100 / 1100 = 1.000
→ 1万円の支出で1万円分の価値（予定通り）
```

ステップ4：完成時総コスト見積り
```
EAC = BAC / CPI = 2000万円 / 1.000 = 2000万円
→ 予算内での完成見込み
```

**評価**：スケジュール遅延が生じているものの、コスト効率は良く、現在のペース（SPI=0.917）が続けば、最終的には予算内で完成する見込みです。ただしスケジュール加速が必要です。

</Callout>

<Callout title="問3：ファンクションポイント法による規模見積り">

次のシステム機能からファンクションポイント法で規模を算出し、開発工数を見積もってください。ただし、開発生産性は 1FP あたり 15 人日と仮定します。

- 入力画面：6項目（複雑度：中、調整係数 4）
- 出力レポート：4種類（複雑度：高、調整係数 7）
- 照会機能：3種類（複雑度：低、調整係数 3）
- マスターファイル：2個（複雑度：中、調整係数 8）

**解答**：

```
入力：6 × 4 = 24FP
出力：4 × 7 = 28FP
照会：3 × 3 = 9FP
内部ファイル：2 × 8 = 16FP
─────────────────
合計：77FP

開発工数 = 77FP × 15人日/FP = 1,155人日
開発期間（概算）= 1,155人日 ÷ 10人/チーム = 115.5人日 ≈ 4〜5ヶ月（チーム規模による）
```

規模は約 77FP で、開発工数は約 1,155 人日です。チーム規模が 5 人なら約 7.7 ヶ月、10 人なら約 4 ヶ月程度の期間を見込みます。

</Callout>
