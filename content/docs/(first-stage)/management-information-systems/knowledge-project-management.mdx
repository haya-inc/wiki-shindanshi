## このページの役割

このページは、`開発を予定どおり進めるための管理` を整理する知識ノードです。`WBS`、`PERT`、`EVM`、`見積り`、`リスク管理` を、計画、日程、進捗、コストの軸で切り分けます。

## 学習のポイント

- まず `作業分解` と `日程管理` を分ける
- 次に `PERT` と `EVM` を、何を測る指標かで切る
- そのうえで `見積り`、`変更管理`、`リスク管理` をまとめて読む

## 試験で何が問われるか

- `WBS` が何を分解する道具かを説明できるか
- `PERT`、`CPM` を、日程計画とクリティカルパスの文脈で読めるか
- `EVM` を、進捗とコストの両面で評価する枠組みとして理解しているか
- リスク管理、見積り、変更管理を、遅延防止や手戻り防止の観点で整理できるか

## 定義と仕組み

### WBS と作業分解

`WBS` は、作業を小さな単位へ分解して、抜け漏れなく管理するための道具です。まず `何の作業があるか` を見える化してから、担当、期間、順序を決めます。

### PERT と CPM

`PERT` は、作業の順序関係と所要時間を図で整理し、全体の日程を考える方法です。`CPM` は、その中でも全体日程を左右する `クリティカルパス` を見つける考え方です。試験では、`どの作業が遅れると全体が遅れるか` を問われやすいです。

### EVM

`EVM` は、`どこまで進んだか` と `いくら使ったか` を同時に見る管理手法です。したがって、単なる `進捗率` ではなく、`進捗とコストのずれ` を読む枠組みとして押さえます。

### 見積り、変更管理、リスク管理

開発では、最初の見積りだけでなく、変更要求やトラブルへの備えも重要です。`見積り` は計画の出発点、`変更管理` は手戻りの抑制、`リスク管理` は起こりそうな問題への先回りと整理します。

## 典型例

| 論点 | 典型的な見分け方 |
| --- | --- |
| `WBS` | まず作業を洗い出す話か |
| `PERT / CPM` | 作業順序と全体日程、クリティカルパスの話か |
| `EVM` | 進捗とコストを同時に評価する話か |
| `リスク管理` | 起こる前に備える話か、起きた後の対処か |

## 典型的なつまずき

- `WBS` を日程表そのものだと思う
- `PERT` と `EVM` をどちらも `進捗管理` として同じものに見てしまう
- `クリティカルパス` を `作業が多い経路` だと誤解する
- 見積り、変更、リスクを別々に暗記して、プロジェクト全体の流れへつなげない

## 問題を解くときの観点

- 問われているのは `作業分解`、`日程計画`、`進捗評価`、`トラブル予防` のどれか
- まず `作業` を見ているのか、`経路` を見ているのか、`コスト` を見ているのか
- 全体日程を左右する作業はどこか
- 問題が起きる前の管理か、進行中の評価か
