# AI 自動改善 タスクリスト

更新日: 2026-03-28

## この文書の役割

この文書は、`中小企業診断士 wiki` を AI が継続的に改善していくための実装タスクを、`小さい作業単位` で管理するための入口です。

目的は、AI が毎回ゼロから考え直すのではなく、`次に何をやるか`、`何を検証するか`、`どこまで進んだら次へ進むか` を固定することです。

## 進め方の前提

- `1バッチ1章` を基本にする
- `ページ数` ではなく `coverage 台帳の状態変化` で前進を測る
- `更新論点` と `安定論点` を分けて扱う
- 本文更新だけで終わらせず、`台帳`、`索引`、`レビュー記録` まで同時に更新する
- 自動化は `低リスクな監査` から始め、`自動執筆` は後段に置く

## フェーズ 1: 監査基盤

- [ ] `docs/wiki-coverage-registry.md` と `content/docs/` のズレを検知する `scripts/check-wiki-coverage.mjs` を作る
- [ ] `docs/agent-handoff.md` の件数、重点領域、次作業が coverage 台帳と一致するか検知する `scripts/check-handoff-summary.mjs` を作る
- [ ] 既存の `pnpm check:wiki` か新しい `pnpm check:wiki-full` に coverage 整合チェックを組み込む
- [ ] `章ページで対応`、`章ページ+演習あり`、`独立ノードあり` の件数を機械集計できるようにする
- [ ] 更新論点の本文にある一次情報 URL と `docs/wiki-freshness-registry.json` の突合結果をレポート化する

完了条件:
`公開面と台帳のズレ` を AI が自動で検知できる

## フェーズ 2: バックログ生成

- [ ] `docs/wiki-completeness-roadmap.md` の優先順位を機械的に読める形へ寄せる
- [ ] `coverage 台帳`、`freshness 台帳`、`review log` から `次の 1 バッチ` を選ぶ `scripts/build-wiki-backlog.mjs` を作る
- [ ] バックログ出力に `対象章`、`対象ページ`、`期待する状態変化`、`必要な更新ファイル` を含める
- [ ] `更新論点` は `要一次情報確認`、`安定論点` は `自動執筆候補` として分岐できるようにする
- [ ] バックログが空なら `未処理なし` ではなく `監査のみ実行` に落ちるようにする

完了条件:
AI が毎回 `どこをやるか` を自由判断せず、同じ基準で次タスクを選べる

## フェーズ 3: 執筆テンプレート化

- [ ] `章ハブ`、`知識ノード`、`出題ノード`、`演習ノード`、`要約` の最小テンプレートを `docs/automation/` 配下に整理する
- [ ] `第2次試験` の `設問型`、`答案骨子`、`事例個別ノード` のテンプレートを整理する
- [ ] `更新論点` 向けに `確認日`、`一次情報`、`本文に入れる要約粒度` の書き分けルールを固定する
- [ ] `meta.json`、`過去問テーマ索引`、`wiki-progress-tracker.md`、`wiki-coverage-registry.md`、`wiki-review-log.md` を同時更新するチェックリストを作る
- [ ] `1バッチ1章` で AI が触ってよいファイル範囲を明文化する

完了条件:
AI がページ種別ごとに迷わず最小差分で追加できる

## フェーズ 4: 自動執筆ワーカー

- [ ] バックログ 1 件を受け取り、対象章だけ編集する `執筆ワーカー` の実行手順を文書化する
- [ ] `章ページで対応` を `独立ノードあり` へ上げる作業を優先するルールを固定する
- [ ] `更新論点` は本文より先に `章ハブ` の確認日と一次情報を更新するルールを入れる
- [ ] `第2次試験` では `設問索引` の列定義を先に固定してから例題を足すルールを入れる
- [ ] `重要用語集` と `重要公式集` は、戻り先が十分に整った後だけ拡張するルールを入れる

完了条件:
AI が 1 回の run で `1 バッチ` を終わらせ、必要な周辺ファイルまで更新できる

## フェーズ 5: 検証ワーカー

- [ ] 変更後に必ず実行するコマンドを固定する
- [ ] 最低ラインを `pnpm check:wiki`、`pnpm lint`、`NEXT_DIST_DIR=.next-check pnpm build` とする
- [ ] `content/docs/` を触った場合だけ、変更ルートの `200` と `h1` を headless browser で確認する
- [ ] 失敗時は `本文差分`、`台帳差分`、`構造違反`、`鮮度違反` に分類して返す
- [ ] 成功時は `何件の coverage が進んだか` を出力する

完了条件:
AI が `書いた` だけで終わらず、`公開できる差分か` まで機械判定できる

## フェーズ 6: 定期実行

- [ ] 毎日走る `監査ジョブ` を設計する
- [ ] 毎週走る `バックログ更新ジョブ` を設計する
- [ ] 毎週走る `安定論点の自動改善ジョブ` を設計する
- [ ] 毎週走る `更新論点の鮮度確認ジョブ` を設計する
- [ ] 結果は `docs/wiki-review-log.md` に追記するか、別の自動実行ログへ集約する

完了条件:
AI が `定期的に監査し、必要時だけ動く` 状態になる

## メモ

- 最初から `自動執筆` を広げすぎない
- 先に `監査` と `次タスク選定` を固める
- `AI が何を書いたか` より `coverage が何件進んだか` を成果にする
- `更新論点` は必ず一次情報確認を通す
