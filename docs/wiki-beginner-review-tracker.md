# 初学者通読レビュー台帳

更新日: 2026-03-30

## 1. この台帳の役割

この台帳は、`初学者が公開導線どおりに順番に読んで、本当に理解し切れるか` を点検するための進行台帳です。

- 作業者が変わっても、`最後にどこまで読んだか`、`次にどこを読むか`、`何を見て判定したか` がすぐ分かるようにする
- 大きな改善ログではなく、`1ページ単位の通読レビュー` を残す
- 共通の学びが出たら、`docs/shindanshi-wiki-strategy.md` と `docs/agent-handoff.md` にも反映する

## 2. 今回の前提

- まず `content/docs` のコア本文 `201ページ` を先に見る
- `content/docs/past-exam-solutions` の `153ページ` は、入口、科目トップ、章ハブ、主要ノードの通読が一巡してから後段で見る
- 1バッチは原則 `1ページ`。入口ページなど比較的重い回でも `2ページ` までに抑える

## 3. 判定ルール

- `通過`: 初学者が、そのページだけで `何のページか`、`なぜ今読むのか`、`次にどこへ行くか` を判断できる
- `要修正`: 定義不足、前提飛躍、比較不足、導線不足のどれかで読者が止まる
- `要分割`: 1ページの役割が広すぎ、追記より分割の方が理解負荷を下げられる

## 4. レビュー時に必ず見る点

- 冒頭で `このページは何のためのページか` が分かるか
- 専門用語の初出に、短い定義や言い換えがあるか
- 結論だけで進まず、`なぜそうなるか` の因果があるか
- 混同しやすい概念を切り分ける軸があるか
- 読み終わったあとに、`次の1ページ` が決まるか

## 8. 今回の学び

1. 入口ページは、回遊先を増やすだけでは弱い。`最初の1ページをどう選ぶか` を先に固定した方が初学者は止まりにくい
2. `1次` と `2次` は受験生には既知に見えても、ホームでは `知識を問う試験` と `事例を読む試験` の違いまで書いた方が導線が安定する
3. 制度ページでは、`全体合格`、`科目合格`、`登録` のような似た言葉の関係を先にほどくと、年度情報の表が読みやすくなる
4. 索引や付録をまとめるページでも、カードの並び順は `利用頻度` ではなく `初回の学習順` に合わせた方が、初学者は迷いにくい
5. 共通プロセス型のハブでは、`全部重要` と並べるだけでは弱い。最初の 3〜5 手を固定した方が、事例別ページへ入る前の迷いを減らせる
6. `ミクロ / マクロ` のように大きく分かれる科目トップでは、用語名だけでなく、`何をどの大きさで見る分野か` を先に書いた方が、次の章を選びやすい
7. `財務・会計` のような積み上げ型の科目トップでは、`記録する → 読む → 判断する → 投資する` の流れを先に示し、代表カードは途中の個別論点ではなく章ハブへ送った方が迷いにくい
8. `企業経営理論` のように複数領域が横並びの科目トップでは、`会社の向き`、`社内実行`、`顧客と施策` のように役割差を一文で切り分け、章ハブの読む順を固定した方が迷いにくい
9. `運営管理` のように工程が前後へつながる科目トップでは、`作る前`、`作りながら改善`、`売場で届ける` の切れ目を先に示し、略語は最初のカードで短く言い換えた方が止まりにくい
10. `経営法務` のように更新論点を含む科目トップでは、`安定論点を読む順` と `更新確認へ戻る順` を明示し、日付入りメモを最初の導線にしない方が初学者は止まりにくい
11. `経営情報システム` のように略語が多い科目トップでは、正式名称を並べるだけでなく `何に使う技術か` を短く補い、`基礎技術 → 実務 → 更新確認` の順を固定した方が止まりにくい
12. `中小企業経営・政策` のように `白書 / 統計 / 補助金 / 支援制度` が同居する科目トップでは、`実態をつかむ章` と `支援策を整理する章` を先に分け、年度更新メモはそのあとに戻す方が初学者は数字や制度名に引っ張られにくい
13. `章ハブ` では、`最初に読む順番` に出てくるページと `カード` と `テーマ別の戻り先` の集合をそろえた方が、初学者が `読む地図` と `戻る地図` を別物として覚えずに済む
14. `マクロ経済学` のように `指標`、`政策モデル`、`理論史` が同居する章ハブでは、`指標 → 需要の中身 → 政策モデル → 理論比較` の順を先に固定した方が、理論名の丸暗記に流れにくい
15. `簿記` のように `転記`、`経過勘定項目`、`伝票制度` などの操作用語が多い章ハブでは、初出で `何をする作業か` を短く言い換えた方が、単語だけを覚える読み方になりにくい
16. `企業会計` のように `B/S`、`P/L`、`CF`、`金商法` の略語と `収益認識` のような独立論点が混在する章ハブでは、正式名を先に出し、章内ノードをカードから落とさない方が初学者の地図が崩れにくい
17. `原価計算` のように `分類軸` と `計算方式` と `差異分析` が同居する章ハブでは、`切り方 → 集計方法 → ずれの理由 → 利益差` の順を先に固定した方が、式だけをつまみ食いする読み方になりにくい
18. `経営分析` のように多数の指標が並ぶ章ハブでは、`儲ける力`、`倒れにくさ`、`資産の回し方` のような役割語でカテゴリを先に言い換えた方が、`高い / 低い` の丸暗記に流れにくい
19. `利益と資金の管理` のように `利益差` と `現金の流れ` が同居する章ハブでは、`予算と実績のズレ`、`営業 / 投資 / 財務の現金区分`、`FCF` の順に固定した方が、損益計算と資金繰りを同じ話として混同しにくい
20. `ファイナンス` のように主本文が長い章でも、冒頭に `主経路` と `周辺ノード` と `戻り先` を分けて置いた方が、途中の理論や比較表を追加しても初学者の地図が崩れにくい
21. `事例Ⅳ` のような事例ハブでは、`設問型 → 個別論点 → 答案骨子` の順を先に固定し、`2次で戻る先` と `1次で戻る先` を同じ表で見せた方が、`どこで解き方を確認し、どこで知識を補うか` を切り分けやすい
22. `事例Ⅰ` のように 2 次の個別ノードがまだ少ない事例ハブでは、`設問型 → 1次の知識ノード → 答案骨子` の主経路を前に出した方が、`何の知識を補ってから短文へ落とすか` が見えやすい
23. `事例Ⅱ` のように `誰に / 何を / どう届けるか` が中核の事例ハブでは、`ターゲット`、`顧客行動`、`価値提案`、`チャネル`、`CRM` の順に 1 次ノードを並べた方が、施策だけを足し算する読み方を防ぎやすい
24. `事例Ⅲ` のように `工程 / QCD / 現場改善` が中核の事例ハブでは、`計画統制`、`動線とレイアウト`、`品質と標準化`、`IE`、`ボトルネック`、`設備 / 外注` の順に 1 次ノードを並べた方が、`効率化する` だけの抽象答案を防ぎやすい
25. `経営戦略論` のように `全社戦略`、`競争戦略`、`経営資源戦略`、`後半論点` が同居する章ハブでは、`会社全体の向き`、`事業への資源配分`、`どう勝つか / どの強みを使うか`、`後半論点へ広げる` の順に固定し、テーマ別の戻り先を持たせた方が戦略の層を混同しにくい
26. `組織論` のように `組織構造`、`組織成立`、`動機づけ`、`リーダーシップ`、`人的資源管理`、`組織文化 / 変革` が同居する章ハブでは、`組織の形`、`組織はなぜ成り立つか`、`人は何で動くか`、`人をどう導くか`、`制度をどう運用するか`、`変化をどう定着させるか` の順に固定した方が、理論家名の丸暗記に流れにくい
