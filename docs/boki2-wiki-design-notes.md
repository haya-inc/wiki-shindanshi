# 簿記2級wiki 設計メモ

中小企業診断士wikiで確立した手法を簿記2級に展開するための設計検討。

## 1. 試験構造の把握（AGENTS.md に書くべき前提）

診断士wikiのAGENTS.mdには「試験の構造」の記述がなかった。
暗黙知として進めたが、簿記2級では試験形式がコンテンツ設計に直結するため、明示が必要。

### 書くべきこと

- 試験形式: 商業簿記3題（60点）+ 工業簿記2題（40点）、70点合格
- 出題形式: 記述式（仕訳・計算・表の穴埋め）。マークシートではない
- ネット試験と統一試験の併存。ネット試験は問題非公開
- 2024年度以降の出題区分表（日商公式）が試験範囲の source of truth
- 到達目標: 「仕訳を書ける」「計算を完走できる」「財務諸表の穴を埋められる」

### 診断士との違いが設計に与える影響

| 観点 | 診断士1次 | 簿記2級 |
|------|-----------|---------|
| 出題形式 | 4択マークシート | 記述式（仕訳・計算） |
| 到達目標 | 選択肢を絞る | 仕訳を書く・計算を完走する |
| 5層の重心 | 概念→公式→計算→誤答→確認 | 仕訳パターン→計算手順→転記→誤答→確認 |
| 過去問 | J-SMECA公式PDF（5年分公開） | 統一試験のみ。ネット試験は非公開 |
| 科目数 | 7科目 | 商業簿記+工業簿記の2分野 |
| 論点の粒度 | 章あたり5-10ノード | 論点が細かい（連結だけで5-8サブ論点） |

## 2. コンテンツ構造の設計

### 案: 2分野 × 章 × ノード

```
content/docs/boki2/
├── commercial/          # 商業簿記
│   ├── index.mdx        # 分野ハブ
│   ├── journal-entries/  # 仕訳パターン（章ハブ + ノード群）
│   ├── financial-statements/  # 財務諸表作成
│   ├── consolidation/   # 連結会計
│   └── ...
├── industrial/          # 工業簿記
│   ├── index.mdx
│   ├── cost-accounting-basics/
│   ├── standard-costing/
│   ├── direct-costing/
│   └── ...
└── reference/           # 過去問マッピング・出題区分表
```

### ノードの粒度ルール

診断士では「1つのノードが試験の1つの大テーマをカバー」だったが、
簿記2級では「1つのノードが1つの仕訳パターンまたは1つの計算手順をカバー」にした方が
通読性と検索性のバランスが良い可能性がある。

- 良い粒度の例: 「有価証券の期末評価」「リース取引の仕訳」「CVP分析」
- 粗すぎる例: 「資産会計」（有形固定資産・無形固定資産・投資有価証券・リースが混在）
- 細かすぎる例: 「満期保有目的債券の償却原価法」（1ページにするほどの量がない）

判断基準: 「このノードだけ読めば、関連する過去問の大問の該当箇所が解ける」サイズ。

## 3. 5層フォーマットの簿記2級版

診断士の5層（概念→公式→計算手順→誤答パターン→確認問題）を
簿記2級の記述式に合わせて調整する。

### 簿記2級の5層

1. **何をする場面か**: この仕訳・計算が必要になる取引のイメージ
2. **仕訳パターン / 計算手順**: 金額つきの仕訳例 or ステップ付き計算手順
3. **転記と集計**: 仕訳から試算表・財務諸表への転記、表の穴埋めパターン
4. **間違えやすいポイント**: 借方貸方の逆転、端数処理、期間按分の罠
5. **確認問題**: 仕訳を書かせる or 金額を計算させるオリジナル問題

### 診断士との違い

- 「公式」の代わりに「仕訳パターン」が中心になる
- 「転記と集計」が新しい層として必要（診断士にはなかった）
- 確認問題は「4択」ではなく「仕訳を書け」「金額を求めよ」形式

## 4. 過去問の著作権整理（最初にやるべき）

### 確認が必要な事項

- 日本商工会議所の過去問著作権ポリシー
- 統一試験の過去問の入手可能範囲（何回分が公開されているか）
- ネット試験は問題非公開 → マッピング対象外
- 市販の予想問題集・模擬問題集の著作権

### 「1+2」方式の適用

診断士と同じ方式が使える可能性が高い:
1. マッピング表: 回次×大問番号×テーマ×対応ノード（問題文なし）
2. オリジナル確認問題: 過去問パターンを参考に自作

ただし、統一試験の公開範囲が限られる場合、マッピング表の実用性が下がる。
その場合は「出題区分表ベースの網羅」をprimaryにし、マッピングはsecondaryにする。

## 5. 参照資料の確保

診断士wikiでは `_book/` に参照テキストがあり、これがdiffの起点だった。
簿記2級でも同等の参照資料が必要。

### 選択肢

- A. 市販テキスト（TAC、ネットスクール等）のPDFを `_book/` に配置
- B. 日商の出題区分表を起点に、公開情報（会計基準、実務指針）から構成
- C. 既存の診断士wikiの財務・会計ノードを参照資料として流用

### 推奨

Bをprimaryにする。出題区分表は日商が公式に公開しており、著作権リスクがない。
Cは「すでに5層で書き直した」財務・会計ノード（簿記・企業会計・原価計算）を
出発点にできるため、ゼロからのスタートよりも大幅に効率的。

## 6. AGENTS.md に書くべき指示の構成

### 必須セクション

```markdown
# AGENTS.md

## 最重要: KISS / YAGNI
（診断士wikiと同一）

## 試験の構造
- 簿記2級の出題形式、配点、合格基準
- 商業簿記と工業簿記の範囲
- ネット試験と統一試験の違い

## コンテンツ構造
- 2分野 × 章 × ノードの3層
- ノードの粒度ルール（上記参照）
- ファイル命名規則

## 5層フォーマット（簿記2級版）
- 5層の定義と各層の書き方
- 仕訳の表記ルール（勘定科目名、金額の単位）
- 計算手順の番号振りルール

## 教材型書き直しパターン
- IS-LMパターンの8要素（診断士wikiから流用）
- 簿記2級特有の調整: 「まずイメージをつかむ」→「どんな取引か」に読み替え

## 過去問対応
- 著作権ルール
- 「1+2」方式
- 出題区分表ベースの網羅

## 品質ゲートと進捗管理
- G1-G6の定義（診断士wikiから流用可）
- review-log / progress-tracker / coverage-registry の運用

## 検証の基本
- pnpm lint + build
- 変更したルートの200確認
```

## 7. 診断士wikiからの具体的な流用候補

### そのまま使えるファイル

- `docs/wiki-review-log.md` の記録フォーマット
- `docs/wiki-progress-tracker.md` の表構造
- `docs/wiki-coverage-registry.md` の台帳構造

### 内容を流用できるページ

診断士wikiの財務・会計セクションと簿記2級は範囲が大きく重なる:

| 診断士wiki | 簿記2級での対応 |
|------------|----------------|
| 簿記の基礎（3ノード） | 商業簿記の基礎部分 |
| 企業会計の基礎（5ノード） | 商業簿記の応用部分 |
| 原価計算（4ノード） | 工業簿記のほぼ全体 |
| 経営分析（3ノード） | 簿記2級の範囲外（一部除く） |
| ファイナンス（4ノード） | 簿記2級の範囲外 |

ただし、簿記2級は診断士より**深い**。連結会計、税効果会計、リース会計などは
診断士レベルの解説では不十分で、仕訳レベルまで掘り下げる必要がある。

## 8. 初手の作業順序

1. 日商の出題区分表を入手し、全論点をリスト化
2. 診断士wikiの財務・会計ノードとのdiffを取り、流用可能範囲を特定
3. コンテンツ構造（ディレクトリ、meta.json）を設計
4. AGENTS.md を作成
5. 最も出題頻度が高い論点（仕訳、連結、標準原価計算あたり）からS群として着手
6. 以降は6ステップ手法に沿って進める

## 9. リスクと対策

| リスク | 対策 |
|--------|------|
| 論点が細かすぎてページ数が爆発する | 粒度ルールを先に決め、「1ノード＝大問の1箇所が解ける」サイズに |
| ネット試験の問題がないためマッピングが不完全 | 出題区分表ベースの網羅をprimaryにする |
| 診断士wikiとの重複管理が煩雑になる | 別リポジトリにするか、モノレポ内で分離するか先に決める |
| 記述式の確認問題はマークシートより作成コストが高い | 仕訳問題は定型化しやすいので、パターンテンプレートを先に作る |
