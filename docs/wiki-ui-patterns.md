# wiki UI パターン

更新日: 2026-03-28

## 目的

この文書は、`/docs` 配下の見た目と情報設計を、ページごとにぶらさず運用するための最小ルールを整理するものです。

## 視覚方針

- 基調: `quiet premium` を目指し、強い演出より `可読性` と `学習速度` を優先する
- 背景: warm paper 系の単色を土台にし、派手な演出ではなく `surface の階層` で奥行きを整理する
- 文字: 見出しは serif、本文は sans、数字や式は mono と役割を分ける
- 密度: `hero → summary → body → next step` を基本にするが、最初の 1 画面は常に役割と導線が先に見えるようにする
- 動き: hover と page load は控えめにし、`読む流れ` を止めない範囲だけに使う
- 運用: 新規ページは個別デザインせず、型に流し込むことで全体の統一感を保つ
- タイトル: docs ページの `breadcrumb` は補助情報として軽く扱い、`h1` はページ内で最も強い見出しとして扱うが、wiki の本文密度を崩さない大きさに抑える
- 節見出し: `h2` と `h3` は本文より一段強い程度に留め、階層差はサイズだけでなく余白と罫線でも見せる
- ヘッダー余白: タイトル直下に説明文がないページでは、アクション行や最初の `h2` の罫線で空白帯を作らない
- 短い分類ブロック: 4件前後の見分け方は `TypeTable` を増やさず、既存の `Cards` を使って一度に見せる
- カード: 横書き本文を細い縦カードへ押し込まない。`stat card` は短い値や状態だけに使い、索引や説明文では使わない

## 採用するページ型

### 1. ハブページ

- 対象: `/docs`、`/docs/second-stage`、`/docs/reference`
- 役割: セクション全体の入口を作る
- 必須要素:
  - 導入ブロック
  - 導線カード
  - 制度確認または運用方針

### 2. 章トップ

- 対象: `/docs/finance-and-accounting` のような科目・章ページ
- 役割: 範囲、優先順位、次に読むページを整理する
- 必須要素:
  - 位置づけ
  - 学習のポイント
  - 優先順位
  - 関連ページ

### 3. 事例ハブ

- 対象: `/docs/second-stage/case-1-organization-and-hr` のような事例別トップページ
- 役割: 1事例の入口を作り、`関連1次科目` と `分解ページ` への往復を固定する
- 必須要素:
  - この事例の位置づけ
  - この wiki で扱う範囲
  - 関連が強い1次科目
  - この事例の分解ページ
  - 学習のポイント
  - 典型的なつまずき
  - 次に読むとよいページ

### 4. 知識ノード

- 対象: `/docs/second-stage/reading-case-materials` のような個別論点ページ
- 役割: 1論点を短く再利用可能な形で整理する
- 必須要素:
  - このページの役割
  - 学習のポイント
  - 典型的なつまずき
  - 関連ページ

### 5. 更新参照ページ

- 対象: `/docs/reference/exam-guide` のような制度変更や年度差を扱う参照ページ
- 役割: `確認日つきの最新情報` と `その時点での読み方` を本文から分離して保つ
- 必須要素:
  - このページの前提
  - 確認日時つきの最新更新点
  - 確認日時つきの読み方
  - この wiki での運用ルール
  - 公式情報

### 6. 索引・付録ページ

- 対象: `/docs/reference/past-question-theme-index`、`/docs/reference/important-formulas` のような道具ページ
- 役割: 説明より `選ぶ / 確認する / 戻る` を最短で行えるようにする
- 必須要素:
  - 通常のタイトルルール
  - 使い方を 1 ブロックで示す導入
  - すぐ使えるジャンプ導線または索引
  - 本文の主役になる表や一覧
- 禁止事項:
  - 右カラムの細い stat card
  - 長い hero 説明
  - 最初の 1 画面を装飾で埋める構成

### 7. 演習ページ

- 対象: `/docs/**/practice-*` のような演習ノード
- 役割: 読ませるより `すぐ解く / すぐ確かめる` を優先する
- 必須要素:
  - 通常のタイトルルール
  - このページの役割
  - 問題または確認ポイント
  - 自力確認しやすい導線
- 禁止事項:
  - 表紙のような大見出し
  - 説明が長い hero
  - 横書き本文を細い補助カードへ押し込む構成

## 事例ページの最小テンプレ

### 事例ハブ

- `## この事例の位置づけ`
- `## この wiki で扱う範囲`
- `## 関連が強い1次科目`
- `## この事例の分解ページ`
- `## 学習のポイント`
- `## 典型的なつまずき`
- `## 次に読むとよいページ`

### 設問型ページ

- `## このページの役割`
- `## まず固定する視点`
- `## 設問型の全体像`
- `## よくある誤答`
- `## 次に読むページ`

### 答案骨子ページ

- `## このページの役割`
- `## 解答を作る順` または `## 先に決める得点計画`
- `## レイヤー別の答案骨子` または `## 設問型別の答案骨子`
- `## 失点を避ける確認`
- `## 関連ページ`

### 事例個別ノード

- `## このページの役割`
- `## 先に固定すること`
- `## 解く順` または `## ○○型の処理順`
- `## 典型例題`
- `## 誤答例と直し方`
- `## よくある失点`
- `## 次に読むページ`

## 実装場所

- 共通スタイル: `app/globals.css`
- `/docs` トップ: `content/docs/index.mdx`
- ハブページ: `content/docs/*/index.mdx`
- 事例ハブ: `content/docs/second-stage/case-*-*.mdx`
- 章トップ・知識ノード: `content/docs/**/*.mdx`
- 更新参照ページ: `content/docs/reference/*.mdx`
- 軽い構造チェック: `scripts/check-doc-page-structure.mjs`

## 運用メモ

- tracked page では、`wiki-progress-tracker` の品質ゲートに合わせて `G1`、`G2`、`G3` の見出しを保つ
- 制度や日程のような更新論点は、本文に固定せず参照資料へ寄せる
- 見た目の改善だけでなく、`次にどこへ進むか` がすぐ分かる導線を優先する
- 配色トークンは `app/globals.css` の `--color-fd-*` を基準にし、`--background` や `--accent-strong` などの独自変数はレイアウトと装飾の別名にとどめる
- ボタンの共通 class は [components/ui/button.tsx](/Users/yhay81/ghq/github.com/haya-inc/shindanshi/components/ui/button.tsx) の `color` と `size` だけを正とし、同義の引数名を増やさない
- ボタンの丸みや余白は呼び出し側で足さず、見た目の差分はまず `size` に寄せる
- カード状の面と feedback 用 textarea は、見た目を呼び出し側へ再記述せず [components/ui/surface.ts](/Users/yhay81/ghq/github.com/haya-inc/shindanshi/components/ui/surface.ts) の class と `textareaVariants` を先に見る
- `content/docs/**/meta.json` では、必要な箇所だけ `---区切り---` を使って sidebar の塊を明確にする
- docs 内の内部リンクは、ページ移動や階層変更に追従しやすいように相対パスを優先する
- `Steps` は手順を順序つきで読ませたいときだけ使い、単なる装飾には使わない
- `TypeTable` は確認項目や見分け方の軸を固定したいページに限定して使う

## 品質ゲート

- `pnpm check:wiki` で鮮度台帳とページ構造を同時に確認する
- `content/docs/second-stage/case-*.mdx` は、型ごとの必須見出しを満たしているか確認する
- `content/docs/reference/exam-guide.mdx` は、更新参照ページの基準形として確認する
- `scripts/check-doc-page-structure.mjs` の型定義は、この文書の `事例ページの最小テンプレ` と揃えて更新する

## UI 確認

- 実行コマンド: `pnpm check:docs-ui`
- 確認対象は `tests/docs-ui.spec.ts` を正とする
- 代表ルートの件数は固定値で管理せず、`tests/docs-ui.spec.ts` の `desktopRouteGroups` を正とする
- `mobile`、`dark mode`、`AI に質問` などの補助確認ルートも、同じファイルの `uiScenarioRoutes` を正とする
- desktop 対象は次の型を代表ルート込みで確認する:
  - ホーム、1次トップ、2次トップ、参照トップ
  - 索引・付録ページ
  - 主要な科目ハブと章トップ
  - 1次の知識ノード、演習ノード、更新ノード
  - 2次の知識ノード、設問型ページ、答案骨子ページ、事例Ⅳ個別ノード
- 追加の操作性確認:
  - mobile でハブページが横スクロールしないこと
  - dark mode で代表ページが崩れないこと
  - keyboard focus が見えること
  - `AI に質問` パネルを開けること
  - representative page に `complementary` と `article` の landmark が存在すること
  - 代表的な本文色の contrast が保たれていること
