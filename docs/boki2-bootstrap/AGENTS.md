# AGENTS.md

## 最重要: KISS / YAGNI

- **KISS**: まず最も単純な解で実装する。賢い抽象よりも「誰でも読めるコード」。
- **YAGNI**: "必要になるまで" 追加しない。将来要件のためのフックや設定や分岐は避ける。
- **単一経路**: 機能は可能な限り一つのコードパスで扱う。条件分岐や設定フラグを増やさない。
- **既存再利用**: 既存のユーティリティや依存関係を流用し、新規レイヤを増やさない。
- **認知負荷を下げる**: 一貫した命名、小さな関数、局所的に理解できる依存関係、最小の設定を優先する。
- **証拠駆動**: 追加前に失敗例、テスト、ユーザー要望を確認し、最小差分で解決する。
- **日本語ドキュメント**: コメント、docstring、ドキュメントは日本語で書く。
- **ドキュメント更新**: コード変更に合わせて、必要十分なドキュメントを更新する。

## シンプルさチェック

- 3 分で概要が伝わるか。
- 読む人が迷う分岐やフラグはないか。
- 既存の仕組みを再利用できないか。
- 引数や設定は最小か。既定値で動くか。
- ドキュメントは必要十分か。将来の仮定を書きすぎていないか。

## 技術基盤

- このリポジトリは `pnpm`、`Next.js 16 App Router`、`Fumadocs` を前提にする。
- Next.js 固有の実装前に、関連する公式ドキュメントを確認する。
- `Cache Components` を前提に設計し、旧来のパターンを新規採用しない。
- `middleware.ts` ではなく `proxy.ts` を使う。

## 試験の構造

このwikiは **日商簿記検定2級** の合格を目標とする学習wikiである。

### 基本情報

- 試験科目: 商業簿記 + 工業簿記（原価計算含む）
- 試験時間: 90分
- 満点: 100点（商業簿記60点 + 工業簿記40点）
- 合格基準: 70点以上（科目別の足切りなし）
- 大問構成: 最大5題（商業簿記3題 + 工業簿記2題）
- 出題形式: **記述式**（仕訳・計算・表の穴埋め）。マークシートではない。
- 受験形式: 統一試験（年3回：6月・11月・2月）+ ネット試験（CBT、随時）

### 出題区分表

- 2026年度は2022年度版の出題区分表を適用（2021年12月改定）
- 2027年度から改定予定（手形の削除、定率法・除却廃棄の追加など）
- source of truth: [日商公式 出題区分表](https://www.kentei.ne.jp/bookkeeping/exam-list)
- 商業簿記PDF: https://www.kentei.ne.jp/wp/wp-content/uploads/2024/12/shogyouboki_kubun.pdf
- 工業簿記PDF: https://www.kentei.ne.jp/wp/wp-content/uploads/2021/12/2022_kogen.pdf

### 到達目標

このwikiの到達目標は「仕訳を書ける・計算を完走できる・表の穴を埋められる」である。
診断士試験のような「選択肢を絞る」能力ではなく、「正しい答えを自力で導く」能力を育てる。

### 過去問の扱い

- 日商簿記2級・3級の過去問は**非公開**（1級のみ公式公開）
- ネット試験の問題も非公開
- 日商はサンプル問題を公開している: https://www.kentei.ne.jp/bookkeeping/sample/sample-two
- 著作権: サンプル問題・模擬問題の著作権は日本商工会議所に帰属、無断転載禁止
- **wiki方針**: 過去問マッピングは作成しない（非公開のため）。代わりに出題区分表ベースの完全網羅を目指す。サンプル問題の傾向分析は行うが、問題文は転載しない。確認問題は全てオリジナルで作成する。

### ノードの粒度

1ノード = **「その論点の仕訳を書ける or 計算を完走できる」サイズ**

- 良い粒度の例: 「有価証券の期末評価」「リース取引の仕訳」「標準原価差異分析」「連結：未実現利益の消去」
- 粗すぎる例: 「資産会計全般」（有形・無形・投資が混在して通読困難）
- 細かすぎる例: 「満期保有目的債券の定額法」（1ページに収まる量がない → 投資その他の資産ノードに含める）

判断に迷ったら「このノードだけ読めば、関連する大問の該当箇所が解ける」かどうかで決める。

### ノードの種類

- **ノード**: 論点の解説。5層フォーマットで書く。
- **出題ノード**: 大問の解き方（仕訳問題、精算表、財務諸表作成など形式別）。
- **演習ノード**: オリジナルの確認問題。仕訳・計算・穴埋め形式。
- **要点整理**: 章の比較表・チェックリスト。

## 5層フォーマット（簿記2級版）

すべてのノードは以下の5層を揃える。

### 1. どんな場面か

この仕訳・計算が必要になる取引のイメージを、日常的な具体例で説明する。
「なぜこの処理が必要なのか」を先に伝える。

### 2. 仕訳パターン / 計算手順

金額つきの仕訳例 or 番号付きの計算ステップ。
仕訳は `借方 / 貸方` の表形式で示し、勘定科目と金額を明記する。

### 3. 転記と集計

仕訳から試算表・精算表・財務諸表への転記パターン。
「この仕訳は精算表のどの欄に現れるか」「P/Lのどの段階に影響するか」を示す。

### 4. 間違えやすいポイント

借方貸方の逆転、端数処理、期間按分、勘定科目の選択ミスなど。
「こう間違える人が多い → 正しくはこう」のペアで示す。

### 5. 確認問題

仕訳を書かせる or 金額を計算させるオリジナル問題。
4択ではなく「仕訳を示せ」「金額を求めよ」形式。答えはAccordionで隠す。

## 教材型書き直しパターン

ノードは以下の8要素で構成する（診断士wikiのIS-LMパターンを簿記2級向けに調整）。

1. **前提知識Callout** → このページを読む前に必要なノードへのリンク
2. **どんな取引か** → 日常の具体例で取引のイメージを作る（「なぜこの処理が必要か」）
3. **出題での位置づけCallout** → サンプル問題・頻出テーマとの対応を示す
4. **仕訳の前に「なぜこの勘定科目か」** → 仕訳パターンを出す前に理由を説明する
5. **金額つき仕訳例** → 具体的な金額で仕訳を完走させる
6. **間違えやすいポイント** → 典型的なミスを先回りで提示
7. **AccordionでQ&A** → 初学者が「なぜ？」と思う箇所への回答
8. **読む順序ガイド** → prev/nextリンクで章内の通読順を案内

## MDXの技術ルール

- **数式**: inline `$...$` は使わない。`$$` ブロック数式 + backtick code に統一する（acornパーサー衝突回避）
- **仕訳表記**: 仕訳はMarkdownテーブルで表現する:
  ```
  | 借方科目 | 金額 | 貸方科目 | 金額 |
  |----------|------|----------|------|
  | 仕入 | 100,000 | 買掛金 | 100,000 |
  ```
- **コンポーネント**: `<Accordions>`, `<Accordion>`, `<Callout>`, `<Cards>`, `<Card>`, `<Steps>`, `<Step>` を使う
- **パス名に括弧を含めない**: ディレクトリ名はハイフン区切りの英数字のみ

## 6ステップ工程

1科目を完全網羅するための確立済み工程（診断士wiki経済学で実証）。

1. **出題区分表diff**: 出題区分表の全細目と既存ノードを突合し、欠落・浅い論点を洗い出す
2. **S/A/B分類**: 出題頻度（サンプル問題・市販問題集の傾向）× 内容ギャップで優先度を3段階に分ける
3. **5層追記**: 5層フォーマット（場面→仕訳→転記→誤答→確認問題）で追記
4. **教材型書き直し**: 8要素パターンでチートシート型→通読型に変換
5. **サンプル問題突合**: 公開サンプル問題の全問を1問ずつwikiノードに対応づけ、カバレッジ確認
6. **ナビ実装**: 各ノードに出題頻度Callout + 出題区分表マップの対応列を整備

### 工程の順序の注意

- ステップ3→4は分けて進める。先に5層で内容を揃えてから教材型に書き直す
- ステップ4と5は同時進行できる。書き直し中にサンプル問題を意識するとギャップに気づく
- ステップ6はステップ4と同時にCalloutを入れると漏れない
